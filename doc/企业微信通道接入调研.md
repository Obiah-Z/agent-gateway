# 企业微信通道接入调研

## 目标

本次调研目标是评估如何把企业微信作为新的消息通道接入 Gateway。接入完成后，企业微信用户可以通过自建应用向 Gateway 发送消息，Gateway 将消息转换为统一的入站任务，经过路由、Agent 执行、可靠投递后，再通过企业微信应用消息接口回复给用户。

从现有架构看，企业微信不需要单独改造 Agent 执行内核。它应该作为一个新的 `Channel` 实现接入，与飞书通道类似，只负责平台协议适配：验签、解密、消息解析、用户标识映射、主动发送和错误归类。

## 当前实现状态

当前 Gateway 已完成企业微信自建应用的文本对话闭环：

| 能力 | 状态 | 说明 |
| --- | --- | --- |
| 回调 URL 验证 | 已实现 | 支持 `msg_signature`、`timestamp`、`nonce`、`echostr` 校验与解密 |
| 文本消息入站 | 已实现 | 支持企业微信加密 XML POST 回调，转换为 `InboundMessage` |
| 入站去重 | 已实现 | 基于 `MsgId` 生成 `wework:{account_id}:msg:{MsgId}` 幂等 key |
| 文本消息出站 | 已实现 | 支持 `gettoken` 获取 `access_token`，调用应用消息 `message/send` |
| token 过期重试 | 已实现 | 对常见 token 失效错误刷新 token 后重试一次 |
| 图片、文件、事件 | 待扩展 | 当前会忽略非文本回调，不进入 Agent 链路 |
| 群聊高级能力 | 待扩展 | 当前按企业微信用户 ID 做单聊映射 |

## 与现有 Gateway 架构的对应关系

Gateway 当前通道层以 `ChannelAccount` 和 `Channel` 为核心抽象。`config/channels.json` 负责声明通道账号，`.env` 负责保存密钥，`load_channel_accounts()` 会把 `token_env` 和 `*_env` 形式的配置解析成运行时配置。

企业微信可以复用这一套模型：

| Gateway 概念 | 企业微信对应内容 |
| --- | --- |
| `channel` | `wework` |
| `account_id` | 企业微信应用账号标识，例如 `wework-main` |
| `token` | 回调 Token，用于验签 |
| `config.corp_id` | 企业 ID `CorpID` |
| `config.agent_id` | 自建应用 `AgentId` |
| `config.secret` | 自建应用 `Secret` |
| `config.encoding_aes_key` | 回调消息加解密密钥 |
| `InboundMessage.sender_id` | 企业微信 `FromUserName` / 用户 ID |
| `OutboundMessage.to` | 企业微信 `touser` |

企业微信通道的核心工作不是改变业务流程，而是把企业微信协议转换为 Gateway 已有的统一消息模型。

## 企业微信接入模式

建议采用“自建应用 + 回调模式 + 应用消息 API”的方式接入。

入站方向：

企业微信后台配置回调 URL、Token 和 EncodingAESKey。用户给自建应用发消息后，企业微信向 Gateway 的 webhook 地址推送加密 XML 消息。Gateway 完成验签、解密和 XML 解析后，生成 `InboundMessage`，交给现有入站任务分发链路。

出站方向：

Gateway 回复用户时，通过企业微信应用消息接口发送文本消息。发送前需要使用 `CorpID + Secret` 获取 `access_token`，并缓存到过期前复用。文本能力稳定后，再扩展 Markdown、图文、文件等消息类型。

## 需要准备的 Key 与配置

企业微信后台需要准备以下配置：

| 配置项 | 来源 | 用途 | 是否必需 |
| --- | --- | --- | --- |
| `WEWORK_CORP_ID` | 企业微信管理后台 | 企业唯一 ID，获取 access_token 时使用 | 必需 |
| `WEWORK_AGENT_ID` | 自建应用详情页 | 指定由哪个应用发送消息 | 必需 |
| `WEWORK_SECRET` | 自建应用详情页 | 获取应用级 access_token | 必需 |
| `WEWORK_CALLBACK_TOKEN` | 自建应用接收消息配置 | 校验回调来源和签名 | 必需 |
| `WEWORK_ENCODING_AES_KEY` | 自建应用接收消息配置 | 解密企业微信推送的加密消息 | 必需 |
| `WEWORK_WEBHOOK_PATH` | Gateway 自定义 | 企业微信回调路径，例如 `/webhooks/wework` | 建议配置 |
| `WEWORK_API_BASE` | Gateway 自定义 | 企业微信 API 根地址，默认 `https://qyapi.weixin.qq.com` | 可选 |

企业微信后台还需要确认：

1. 自建应用的可见范围包含测试用户。
2. 回调 URL 可以被企业微信公网访问。
3. 如果企业微信后台或接口要求可信 IP，需要加入 Gateway 出口公网 IP。
4. 如果后续要发送文件，需要补充素材上传相关权限和文件类型限制验证。

## 推荐配置形式

`config/channels.json` 可以新增一个企业微信账号：

```json
{
  "channel": "wework",
  "account_id": "wework-main",
  "label": "企业微信主应用",
  "enabled": true,
  "token_env": "WEWORK_CALLBACK_TOKEN",
  "config": {
    "corp_id_env": "WEWORK_CORP_ID",
    "agent_id_env": "WEWORK_AGENT_ID",
    "secret_env": "WEWORK_SECRET",
    "encoding_aes_key_env": "WEWORK_ENCODING_AES_KEY",
    "webhook_path": "/webhooks/wework",
    "api_base": "https://qyapi.weixin.qq.com"
  }
}
```

`.env.example` 已包含以下占位配置：

```dotenv
# 企业微信自建应用
WEWORK_CORP_ID=
WEWORK_AGENT_ID=
WEWORK_SECRET=

# 企业微信回调配置
WEWORK_CALLBACK_TOKEN=
WEWORK_ENCODING_AES_KEY=
WEWORK_WEBHOOK_PATH=/webhooks/wework
WEWORK_API_BASE=https://qyapi.weixin.qq.com
```

## 入站链路设计

企业微信入站需要处理两类请求。

第一类是 URL 验证请求。企业微信配置回调 URL 时，会携带 `msg_signature`、`timestamp`、`nonce` 和 `echostr` 请求 Gateway。Gateway 需要用 Token、EncodingAESKey 和签名参数验证来源，解密 `echostr` 后返回明文，才能完成后台配置。

第二类是消息与事件推送。企业微信会推送加密 XML，Gateway 需要先验签，再解密 XML，解析出消息类型、发送人、接收应用、消息 ID、创建时间和文本内容。文本消息先进入主流程，图片、文件、事件等类型可以先记录为 ignored 或 unsupported event，避免影响文本链路上线。

建议的入站映射：

| 企业微信字段 | Gateway 字段 |
| --- | --- |
| `FromUserName` | `sender_id` |
| `FromUserName` | `peer_id`，单聊阶段可以与 `sender_id` 相同 |
| `Content` | `text` |
| `MsgId` | `metadata.platform_message_id` |
| `CreateTime` | `metadata.platform_created_at` |
| `MsgType` | `metadata.message_type` |
| `AgentID` | `metadata.agent_id` |

去重 key 建议按优先级选择：

1. 文本消息优先使用 `wework:{account_id}:msg:{MsgId}`。
2. 事件没有 `MsgId` 时，使用 `wework:{account_id}:event:{FromUserName}:{CreateTime}:{Event}:{EventKey}`。
3. URL 验证请求不进入任务链路，只做验证响应。

## 出站链路设计

企业微信出站先支持文本消息。流程为：

1. 检查本地缓存的 `access_token` 是否有效。
2. 如果不存在或即将过期，通过 `CorpID + Secret` 请求企业微信 `gettoken` 接口。
3. 组装应用消息，`OutboundMessage.to` 映射为 `touser`，`agent_id` 映射为 `agentid`。
4. 调用企业微信应用消息发送接口。
5. 根据企业微信错误码判断是否重试、刷新 token 或标记永久失败。

文本消息体形态可以按以下方向适配：

```json
{
  "touser": "USER_ID",
  "msgtype": "text",
  "agentid": 1000002,
  "text": {
    "content": "Gateway 回复内容"
  },
  "safe": 0
}
```

后续可扩展：

| 类型 | 优先级 | 说明 |
| --- | --- | --- |
| text | P0 | 先保证基础对话闭环 |
| markdown | P1 | 适合巡检、报告、结构化输出，但需确认企业微信应用消息支持差异 |
| file | P1 | 用于发送 Gateway 生成的报告附件，需要先上传临时素材 |
| news / mpnews | P2 | 适合卡片化摘要，非首期必需 |

## 实现文件

当前按独立企业微信通道包实现核心协议适配：

```text
agent_gateway/gateways/wework/
  __init__.py
  channel.py
  crypto.py
```

职责划分：

| 文件 | 职责 |
| --- | --- |
| `channel.py` | 实现 `WeWorkChannel`，负责发送、解析已解密消息、token 缓存和错误归类 |
| `crypto.py` | 实现企业微信回调签名校验、AES 解密、URL 验证解密 |
| `feishu/http.py` | 当前复用既有 webhook server 端口，按 path 分流企业微信 GET/POST 回调 |
| `messaging/bootstrap.py` | 注册 `wework` 通道工厂 |
| `app.py` 或 runtime 装配层 | 在启动时挂载企业微信 webhook server |

首期不建议把企业微信逻辑塞进飞书通道。两者都属于 IM 平台，但验签、加密、消息体、发送接口和错误码不同，独立通道更清晰。

## 测试计划

首期至少覆盖以下测试：

1. `channels.json` 中 `*_env` 能正确解析企业微信配置。
2. URL 验证请求能正确验签并返回解密后的 `echostr`。
3. 文本消息回调能转换为 `InboundMessage`。
4. 重复 `MsgId` 不会重复进入任务链路。
5. `send()` 能正确构造企业微信文本消息体。
6. `access_token` 能缓存，并在过期或 token 错误时刷新。
7. 企业微信 API 返回不可恢复错误时，投递链路能标记 permanent failure。

## 风险与注意事项

企业微信接入的主要风险不在 Agent 侧，而在平台协议边界：

回调加密是首要风险。企业微信推送通常是加密 XML，必须正确处理签名、AES 解密、padding、CorpID 校验和 XML 安全解析。建议不要手写复杂密码学细节，优先参考企业微信官方示例或使用成熟实现，并补充回归测试。

重复推送需要特别处理。企业微信、反向代理、Gateway 自身重试和 RabbitMQ 重投都可能让同一条消息多次进入系统。必须在平台消息 ID 层做幂等 key，不能只依赖后面的 session lane。

用户 ID 映射需要提前设计。企业微信的 `FromUserName` 是企业内用户 ID，不等同于手机号或外部联系人 ID。首期可以直接作为 `sender_id`，后续如果要做用户画像或跨平台身份统一，需要单独维护映射表。

文件发送不是首期最小闭环。企业微信发送文件通常要先上传素材再发送消息，和飞书附件发送不是同一套协议。建议文本稳定后再扩展报告附件下载能力。

## 分阶段实施建议

第一阶段完成文本闭环：新增 `wework` 通道、GET URL 验证、POST 文本消息解析、文本主动回复、基础测试。

第二阶段补齐可靠性：接入 Redis 幂等 key、完善错误码分类、access_token 刷新、Dashboard 事件展示和投递失败可观测性。

第三阶段扩展能力：支持 Markdown、文件上传、群聊消息、事件消息、用户信息查询和多企业微信应用账号。

## 结论

企业微信适合按 Gateway 的标准通道模型接入。需要准备的关键配置是 `CorpID`、`AgentId`、`Secret`、回调 `Token`、`EncodingAESKey` 和公网回调 URL。实现上应优先完成自建应用文本消息的入站和出站闭环，再逐步扩展文件、Markdown 和事件处理。

由于企业微信后台配置和部分接口策略可能随企业版本、应用类型和权限范围变化，正式开发前需要在企业微信管理后台按当前应用配置做一次最终校验。
