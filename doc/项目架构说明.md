## 3. `agent_gateway/` 源码目录

这是项目主包，基本可以理解为网关系统本体。

### 3.1 顶层文件

- `agent_gateway/app.py`
  应用装配入口。负责：
  - 加载环境变量和 `GatewaySettings`
  - 初始化 Agent、路由、会话、工具、记忆、技能
  - 组装 Dispatcher、AutonomyRuntime、DeliveryRuntime、MetricsRuntime、AlertsRuntime
  - 启动 WebSocket 控制面、飞书接入、Dashboard 静态服务
  - 提供 CLI / `serve` 启动入口

- `agent_gateway/config.py`
  环境变量与路径总配置定义。核心作用：
  - 定义 `GatewaySettings`
  - 统一读取 `.env`
  - 解析工作区、数据目录、端口、飞书参数、事件保留期、指标采样周期、告警周期等

- `agent_gateway/config_loader.py`
  负责 `config/*.json` 的加载、保存和默认文件生成，例如 Agent 配置、Bindings、Channels、Profiles。

顶层包不再保留 `agents.py`、`router.py`、`models.py`、`ids.py` 这类兼容层入口。
业务代码应直接从 `agent_gateway.core` 或具体子模块导入领域对象，避免继续扩散旧路径依赖。

### 3.2 `core/` 领域层

这个目录放“比较稳定的核心领域模型”，不直接关心具体通道、前端或运行时装配。

- `core/agents.py`
  定义 Agent 管理能力，例如注册 Agent、读取 Agent、列出 Agent。

- `core/models.py`
  定义核心数据结构，例如 Agent 配置、Binding、回复对象、主动任务目标等。

- `core/router.py`
  定义路由绑定表和匹配逻辑，负责把某条入站消息解析到对应 Agent / session。

- `core/ids.py`
  提供 ID 规范化函数，例如 `agent_id` 的清洗、统一格式处理。

### 3.3 `application/` 应用层

这是系统最关键的一层，负责把各个模块串起来，形成真正可运行的业务流程。

- `application/loop.py`
  Agent 单轮执行闭环。负责：
  - 读取会话历史
  - 拼接 prompt / 记忆 / 技能说明
  - 调用模型执行器
  - 写回最新消息历史
  - 记录 `agent.turn.started/completed/failed` 事件

- `application/resilience.py`
  模型调用容灾层。通常负责：
  - 调用 Anthropic 兼容接口
  - 失败重试 / profile 切换
  - 工具调用闭环执行
  - stop_reason 处理

- `application/dispatcher.py`
  核心消息分发器。负责：
  - 接收入站消息
  - 路由到正确 Agent
  - 调用 `AgentLoopRunner`
  - 将结果投递到可靠消息队列
  - 记录 route / delivery / failure 相关事件

- `application/channel_runtime.py`
  通道运行时。负责把外部通道事件接入 Dispatcher，并协调回复发送时机。

- `application/delivery_runtime.py`
  可靠投递运行时。负责：
  - 从本地投递队列读取待发送消息
  - 调用各 channel 的发送逻辑
  - 失败重试
  - 将成功和失败写入事件流

- `application/autonomy.py`
  主动任务运行时。负责：
  - Heartbeat
  - 全局 Cron
  - Agent 局部 Cron
  - 主动触发后台 Agent 轮次

- `application/lanes.py`
  并发车道与命名队列控制，避免同一对象上的任务互相踩踏。

- `application/control_plane.py`
  控制面服务对象。负责提供管理接口，例如：
  - 查看 Agent / Bindings / Channels / Profiles
  - 查看 delivery 状态
  - 查看 events / errors / memories
  - 查看 metrics / alerts
  - 为 WebSocket JSON-RPC 和 Dashboard 提供数据

- `application/agent_manifest.py`
  Agent 配置模板、能力约束、工具能力白名单校验。

- `application/metrics_runtime.py`
  指标采集运行时。定期将队列、车道、Cron、Profile、事件统计等写成指标快照。

- `application/alerts_runtime.py`
  告警运行时。根据指标和规则计算当前是否触发告警，并将告警写入本地存储和消息投递链路。

### 3.4 `channels/` 通道适配层

这个目录负责“发消息”和“接消息”的具体通道实现。

- `channels/base.py`
  通道抽象基类或通用接口定义。

- `channels/manager.py`
  通道管理器，统一管理多个账号、多种 channel 的实例。

- `channels/bootstrap.py`
  根据 `config/channels.json` 构建通道实例。

- `channels/cli.py`
  本地命令行交互通道，适合开发调试。

- `channels/telegram.py`
  Telegram 通道实现。

- `channels/feishu.py`
  飞书通道发送与接收的主适配层。

- `channels/feishu_cards.py`
  飞书卡片消息或富文本渲染辅助。

- `channels/feishu_state.py`
  飞书通道状态管理，例如去重、状态缓存、消息发送辅助状态。

### 3.5 `interfaces/` 外部接口层

这个目录偏“接入协议”，不是业务本身。

#### `interfaces/feishu/`

- `http.py`
  飞书 Webhook HTTP 服务端，实现 challenge、事件接收、签名校验接入。

- `security.py`
  飞书安全校验逻辑，例如签名、时间窗、加解密、合法性判断。

- `long_connection.py`
  飞书长连接模式接入，适合本地开发和无需公网回调的运行模式。

#### `interfaces/websocket/`

- `server.py`
  WebSocket 控制面服务端，实现 JSON-RPC 风格管理接口，供 Dashboard 和外部控制客户端使用。

### 3.6 `delivery/` 可靠消息投递

- `delivery/queue.py`
  本地预写投递队列。核心作用：
  - 消息先入队再发送
  - 持久化待发送项
  - 失败项重试和归档

### 3.7 `sessions/` 会话与上下文

- `sessions/store.py`
  会话消息 JSONL 存储，支持加载、重写、追加历史。

- `sessions/context.py`

### 3.8 `intelligence/` Prompt、记忆、技能

- `intelligence/bootstrap.py`

- `intelligence/memory.py`

- `intelligence/skills.py`

### 3.9 `tools/` 工具系统

- `tools/builtin.py`

- `tools/web_search.py`

### 3.10 `observability/` 可观测性

- `observability/metrics.py`

- `observability/alerts.py`

### 3.11 `monitoring/` 运维面板

### 3.12 `onboarding/` 接入引导

### 3.13 `news/` 新闻采集与摘要

## 4. `config/` 静态配置目录

这个目录放相对稳定、可版本管理的 JSON 配置。

- `config/agents.json`
  Agent 定义清单。包括每个 Agent 的名称、模型、工具能力、提示词策略、记忆策略等。

- `config/bindings.json`
  路由绑定表。决定某个 channel/account/peer/session 的消息进入哪个 Agent。

- `config/channels.json`
  通道账号配置。定义 CLI、Telegram、飞书等账号是否启用、使用什么模式、附带哪些配置。

- `config/profiles.json`
  模型服务 profile 配置。用于主模型、备用模型、不同供应商配置切换。

## 8. 当前项目的分层理解方式

如果从“系统运行链路”看，这个项目大致可以这样理解：

1. 外部消息从 `interfaces/` 或 `channels/` 进入。
2. `dispatcher.py` 根据 `bindings.json` 做路由。
3. `loop.py` 负责一次 Agent 执行闭环。
4. `resilience.py` 处理模型调用、工具调用、容灾切换。
5. 回复先进入 `delivery/queue.py`，再由 `delivery_runtime.py` 异步发送。
6. `autonomy.py` 负责主动任务，不依赖用户显式发消息。
7. `observability/` 记录事件、指标、告警。
8. `monitoring/` 和 `interfaces/websocket/server.py` 把这些运行状态展示到运维面板。

## 11. 总结

这个项目已经不是单纯的聊天机器人 Demo，而是一个逐步成型的 Agent Runtime：

- `agent_gateway/` 是运行内核
- `config/` 是结构化静态配置
- `workspace/` 是智能体内容层
- `data/` 是运行状态和恢复依据
- `tests/` 是回归保护网

如果你后续要继续接手或重构，最推荐的切入顺序是：

1. 先看 `app.py` 了解装配关系。
2. 再看 `dispatcher.py + loop.py + resilience.py` 理解执行主链路。
3. 然后看 `autonomy.py + delivery_runtime.py` 理解主动任务和可靠投递。
4. 最后看 `control_plane.py + observability/* + monitoring/*` 理解运维和排障体系。
