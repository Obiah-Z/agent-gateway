# 20.8 压测执行清单

本文档用于 Phase 20.8 的全方位测试，目标不是单次跑分，而是找出系统的吞吐上限、QPS 边界、尾延迟拐点、队列积压临界点，以及在故障条件下的退化方式。

## 1. 压测目标

- 看系统在不同并发下能承受多大负载。
- 看 `QPS` 和吞吐量能到多少。
- 看 `P95` 延迟在哪个并发点开始明显恶化。
- 看入站队列、投递队列和 RabbitMQ 是否会持续积压。
- 看 Redis、PostgreSQL、RabbitMQ、模型 API、飞书通道出问题时，系统是否还能明确报错并保持可恢复。

## 2. 参数说明

| 参数 | 含义 | 使用场景 |
| --- | --- | --- |
| `--scenario` | 压测场景名 | 决定测本地闭环、投递队列、真实模型还是飞书发送 |
| `--requests` | 请求总数 | 一次压测发多少个样本 |
| `--concurrency` | 并发数 | 同时运行多少个请求，用来找容量边界 |
| `--agent-delay-ms` | 模拟 Agent 延迟 | `mock-local` 场景中模拟模型或业务处理耗时 |
| `--basename` | 报告名前缀 | 生成的 JSON / Markdown 报告文件名会带这个前缀 |
| `--allow-real-external` | 允许真实外部调用 | 真实模型 / 真实飞书场景必须显式开启 |
| `--agent-id` | 指定 Agent | 真实模型压测时指定路由到哪个 Agent |
| `--prompt` | 压测输入文本 | 控制模型行为，避免工具调用干扰纯模型性能测试 |
| `--delivery-delay-ms` | 模拟投递延迟 | 用来观察投递慢时的排队和积压情况 |
| `--session-prefix` | 会话前缀 | 用于区分压测会话和正常会话，方便后续清理 |
| `--feishu-account-id` | 飞书账号 ID | `feishu-send-real` 场景指定发送使用的飞书通道账号 |
| `--feishu-peer-id` | 飞书接收方 ID | `feishu-send-real` 场景指定消息发给谁 |
| `--message-text` | 飞书发送内容 | `feishu-send-real` 场景指定测试消息内容 |
| `--env-file` | 环境变量文件 | 指定从哪个 `.env` 文件读取模型、飞书和中间件配置 |
| `--report-dir` | 报告目录 | 指定压测 JSON / Markdown 报告输出目录 |
| `--work-dir` | 临时工作目录 | 投递压测中隔离临时状态和队列文件 |
| `--rabbitmq-url` | RabbitMQ 地址 | `delivery-rabbitmq` 场景指定 broker 地址 |
| `--inbound-rabbitmq-partitions` | 入站分区数 | `inbound-rabbitmq` 场景控制 RabbitMQ partition queue 数量 |
| `--inbound-session-count` | 入站会话数量 | `inbound-rabbitmq` 场景控制 session 分布；数量越少越容易形成热点 session |
| `--inbound-rabbitmq-prefetch` | 入站预取数量 | `inbound-rabbitmq` 场景控制消费者预取，默认 1，便于观察 session lane 串行语义 |
| `--inbound-lane-mode` | 入站 lane 验证模式 | `local` 默认启用本地 lane ownership 探针，`redis` 使用真实 Redis lane ownership，`off` 只测 broker/worker 对照 |
| `--redis-url` | Redis 地址 | `--inbound-lane-mode redis` 时用于连接真实 Redis |
| `--inbound-lane-ttl-seconds` | lane 租约 TTL | Redis lane owner 的过期时间，用于验证 worker 崩溃后的接管窗口 |
| `--inbound-lane-namespace` | lane key 命名空间 | Redis lane key 前缀，建议压测使用独立 namespace，避免影响运行中网关 |

## 3. QPS 为什么要分层测试

不能只用真实模型链路测试系统 QPS。真实链路中，模型调用通常是最慢的一步，端到端耗时会被模型供应商、网络、上下文长度和限流策略主导。

完整链路大致是：

```text
入站消息
  -> ChannelRuntime
  -> 路由匹配
  -> lane / task 调度
  -> Prompt 和上下文装配
  -> 模型 API 调用
  -> 工具调用和结果回填
  -> 会话持久化
  -> 可靠投递队列
  -> RabbitMQ / 本地队列
  -> DeliveryRuntime
  -> 飞书 / CLI / Telegram 出站
```

其中 `模型 API 调用` 可能从几百毫秒到几十秒不等。如果直接用真实模型端到端测 QPS，得到的主要是“模型 API + 网络 + 外部限流”的能力，不是 Gateway 自身的调度能力。

因此需要至少给出三组指标：

| 指标 | 推荐场景 | 说明 |
| --- | --- | --- |
| 网关自身 QPS | `mock-local` | 绕开真实模型和外部平台，测 ChannelRuntime、Dispatcher、lane 和状态写入能力 |
| 入站任务 QPS | `inbound-rabbitmq` | 测 RabbitMQ 入站分区、task_id 预占、session lane ownership 和 TaskWorkerRuntime 消费能力 |
| 投递 QPS | `delivery-local` / `delivery-rabbitmq` | 测可靠投递队列、RabbitMQ、ack、retry 和 worker 消费能力 |
| 真实业务 QPS | `model-real` + `feishu-send-real` | 测真实模型和真实飞书通道下的用户体验 |

最终汇报时不要只说一个 QPS。更准确的说法是：

- Gateway 自身调度 QPS 是多少。
- RabbitMQ 投递链路 QPS 是多少。
- 真实模型链路端到端 QPS 是多少。
- 真实业务瓶颈主要卡在模型、投递、中间件还是外部通道。

## 4. 推荐执行顺序

### 4.1 本地闭环

先测网关自身调度能力，不依赖真实模型和外部平台。

```bash
python scripts/load_test_gateway.py --scenario mock-local --requests 200 --concurrency 4 --basename bench-mock-4
python scripts/load_test_gateway.py --scenario mock-local --requests 500 --concurrency 8 --basename bench-mock-8
python scripts/load_test_gateway.py --scenario mock-local --requests 1000 --concurrency 16 --basename bench-mock-16
```

看什么：

- 吞吐量是否随并发提升而提升。
- `E2E P95` 是否在高并发下明显恶化。
- 是否出现入站 lane 积压。

模拟模型耗时的版本：

```bash
python scripts/load_test_gateway.py --scenario mock-local --requests 200 --concurrency 4 --agent-delay-ms 500 --basename bench-mock-agent-delay-500ms
python scripts/load_test_gateway.py --scenario mock-local --requests 200 --concurrency 8 --agent-delay-ms 1000 --basename bench-mock-agent-delay-1000ms
```

这组命令用于回答：如果每个 Agent 任务平均耗时 500ms 或 1000ms，当前 lane 和并发配置能承受多少请求。

### 4.2 投递队列

再测可靠投递链路，区分本地队列和 RabbitMQ 的损耗。

```bash
python scripts/load_test_gateway.py --scenario delivery-local --requests 200 --concurrency 1 --delivery-delay-ms 0 --basename bench-delivery-local
python scripts/load_test_gateway.py --scenario delivery-rabbitmq --requests 200 --concurrency 4 --delivery-delay-ms 0 --basename bench-delivery-rabbitmq
```

看什么：

- 本地队列和 RabbitMQ 的吞吐差异。
- `Delivery P95` 是否上升。
- 队列积压是否持续增长。
- worker 是否出现 ack / retry / dead letter 异常。

模拟慢投递的版本：

```bash
python scripts/load_test_gateway.py --scenario delivery-rabbitmq --requests 200 --concurrency 4 --delivery-delay-ms 500 --basename bench-delivery-rabbitmq-delay-500ms
python scripts/load_test_gateway.py --scenario delivery-rabbitmq --requests 200 --concurrency 8 --delivery-delay-ms 1000 --basename bench-delivery-rabbitmq-delay-1000ms
```

这组命令用于回答：如果外部通道发送变慢，RabbitMQ 和 DeliveryRuntime 会从哪个并发点开始积压。

### 4.3 入站 RabbitMQ 与分布式 lane

再测入站任务化链路，验证 RabbitMQ 分区、task_id 精确预占和 TaskWorkerRuntime worker 池吞吐。

```bash
python scripts/load_test_gateway.py --scenario inbound-rabbitmq --requests 200 --concurrency 4 --agent-delay-ms 0 --inbound-session-count 32 --inbound-rabbitmq-partitions 8 --basename bench-inbound-rabbitmq-c4
python scripts/load_test_gateway.py --scenario inbound-rabbitmq --requests 500 --concurrency 8 --agent-delay-ms 0 --inbound-session-count 64 --inbound-rabbitmq-partitions 16 --basename bench-inbound-rabbitmq-c8
```

真实 Redis lane 版本：

```bash
python scripts/load_test_gateway.py \
  --scenario inbound-rabbitmq \
  --requests 200 \
  --concurrency 8 \
  --agent-delay-ms 10 \
  --inbound-session-count 64 \
  --inbound-rabbitmq-partitions 16 \
  --inbound-lane-mode redis \
  --redis-url redis://127.0.0.1:6379/0 \
  --inbound-lane-ttl-seconds 30 \
  --inbound-lane-namespace gateway:load-test:lane \
  --basename bench-inbound-rabbitmq-redis-lane
```

日常验收 smoke 版本：

```bash
python scripts/smoke_distributed_lane.py \
  --requests 8 \
  --concurrency 4 \
  --session-count 4 \
  --rabbitmq-partitions 4 \
  --redis-url redis://127.0.0.1:6379/0 \
  --basename smoke-distributed-lane
```

这条命令会强校验：成功数等于请求数、`lane_mode=redis`、`max_same_session_concurrency=1`、RabbitMQ 消费后积压为 0、DLQ 为 0、Redis health 正常。它适合每次修改分布式入站链路后做快速验收。

看什么：

- `broker_after_publish.messages` 是否等于请求数。
- `broker_after_consume.messages` 是否回到 0。
- `max_inbound_backlog` 是否随请求数升高持续堆积。
- `partitions_seen` 是否接近 `--inbound-rabbitmq-partitions`，如果很低说明 session 分布集中。
- `max_same_session_concurrency` 是否始终为 1；如果大于 1，说明 session lane 串行保护失效。
- `max_active_lanes` 是否随 session 数增加而升高；如果始终很低，说明热点 session 或分区分布限制了并行度。
- `redis_health` 是否显示 Redis 可用；如果 Redis 不可用，`redis` 模式不能代表真实分布式互斥能力。
- `lane_acquire_conflicts` 是否可解释；热点 session 或 TTL 设置过短都可能导致冲突增加。
- `E2E P95` 是否随 `--concurrency` 增加而下降或到达平台上限。

热点 session 版本：

```bash
python scripts/load_test_gateway.py --scenario inbound-rabbitmq --requests 200 --concurrency 8 --agent-delay-ms 100 --inbound-session-count 1 --inbound-rabbitmq-partitions 8 --basename bench-inbound-rabbitmq-hot-session
python scripts/load_test_gateway.py --scenario inbound-rabbitmq --requests 200 --concurrency 8 --agent-delay-ms 100 --inbound-session-count 64 --inbound-rabbitmq-partitions 8 --basename bench-inbound-rabbitmq-many-sessions
```

这组命令用于回答：同样的 worker 池和 RabbitMQ 分区下，单热点 session 与多 session 分布对吞吐和 P95 的影响有多大。

注意：

- 该场景不调用真实模型，也不发送飞书消息。
- 任务事实状态使用脚本内 `InMemoryTaskBackend`，不访问真实 PostgreSQL。
- 默认 `--inbound-lane-mode local` 会启用本地 lane ownership 探针，验证同 session 串行；`redis` 会使用真实 Redis lane owner，更接近分布式 worker 部署；如需只测 broker/worker 成本，可设置 `--inbound-lane-mode off`。
- 每个压测 worker 会使用独立 RabbitMQ broker 连接，避免多个 worker 并发共享同一个 pika `BlockingChannel`。
- 它验证的是入站 broker 分发、lane ownership 和 worker 消费成本，不等同于真实 AgentLoop 成本。

本机 smoke：

```text
requests=4 concurrency=2 agent_delay_ms=1 session_count=2 partitions=2
success=4 failed=0 throughput_rps≈159 e2e_p95_ms≈26.74
max_active_lanes=2 max_same_session_concurrency=1 lane_acquire_conflicts=2
broker_after_consume.messages=0 dead_letter_messages=0
```

本机 Redis lane smoke：

```text
requests=4 concurrency=2 agent_delay_ms=1 session_count=2 partitions=2 lane_mode=redis
success=4 failed=0 throughput_rps≈105.288 e2e_p95_ms≈37.914
max_same_session_concurrency=1 broker_after_consume.messages=0
```

### 4.4 真实模型

再测真实 Agent 执行链路，观察上下文装配、模型调用和工具链是否成为瓶颈。

```bash
python scripts/load_test_gateway.py --scenario model-real --allow-real-external --requests 20 --concurrency 1 --agent-id main --session-prefix loadtest-model --prompt "请用一句中文回复 pong，不要调用工具。" --basename bench-model-real-c1
python scripts/load_test_gateway.py --scenario model-real --allow-real-external --requests 20 --concurrency 2 --agent-id main --session-prefix loadtest-model --prompt "请用一句中文回复 pong，不要调用工具。" --basename bench-model-real-c2
python scripts/load_test_gateway.py --scenario model-real --allow-real-external --requests 30 --concurrency 4 --agent-id main --session-prefix loadtest-model --prompt "请用一句中文回复 pong，不要调用工具。" --basename bench-model-real-c4
```

看什么：

- 模型 API 的真实延迟。
- `Agent P95` 是否高于本地闭环场景。
- 工具调用是否被意外触发。
- 错误是否能明确落到模型配置、上下文或外部 API。

注意：

- 真实模型压测会产生真实 API 消耗。
- 并发不要一开始拉太高，避免触发模型供应商限流。
- `--prompt` 明确要求不要调用工具，是为了让结果更接近纯模型链路。
- 如果要测工具调用性能，应另起专项压测，不要和纯模型 QPS 混在一起。

### 4.5 真实飞书发送

再测出站通道，观察发送耗时和外部平台限制。

```bash
python scripts/load_test_gateway.py --scenario feishu-send-real --allow-real-external --requests 20 --concurrency 1 --message-text "AI Agent Gateway 飞书发送压测。" --basename bench-feishu-send-c1
python scripts/load_test_gateway.py --scenario feishu-send-real --allow-real-external --requests 20 --concurrency 2 --message-text "AI Agent Gateway 飞书发送压测。" --basename bench-feishu-send-c2
```

看什么：

- 飞书发送延迟。
- 是否触发限流、权限、验签或加密问题。
- 投递失败后是否能重试并落盘。

如果需要指定账号和接收方：

```bash
python scripts/load_test_gateway.py --scenario feishu-send-real --allow-real-external --requests 20 --concurrency 1 --feishu-account-id feishu-main --feishu-peer-id <你的_open_id_or_chat_id> --message-text "AI Agent Gateway 飞书发送压测。" --basename bench-feishu-send-main
```

注意：

- 这会发送真实飞书消息。
- 不建议高并发轰炸飞书接口，容易触发平台限流或打扰用户。
- 当前脚本支持真实飞书发送压测，但还没有独立的真实飞书 Webhook 入站压测场景。

### 4.6 当前完整链路测试方法

当前脚本还没有单条命令覆盖“真实飞书入站 -> Gateway -> 真实模型 -> 可靠投递 -> 飞书出站”的完整链路。因此现阶段完整链路用分段方式覆盖：

1. 用 `model-real` 覆盖真实 Agent 执行链路。
2. 用 `feishu-send-real` 覆盖真实飞书出站链路。
3. 用实际飞书手动发消息，观察 Dashboard 事件流，验证真实入站链路。

分段完整链路命令：

```bash
python scripts/load_test_gateway.py --scenario model-real --allow-real-external --requests 20 --concurrency 1 --agent-id main --session-prefix e2e-model --prompt "请用一句中文回复 pong，不要调用工具。" --basename e2e-model-real-c1
python scripts/load_test_gateway.py --scenario feishu-send-real --allow-real-external --requests 20 --concurrency 1 --message-text "AI Agent Gateway 端到端出站压测。" --basename e2e-feishu-send-c1
python scripts/build_capacity_baseline.py
```

人工验证真实入站：

```text
1. 启动 Gateway。
2. 从飞书给机器人发送一条短消息，例如：hello。
3. 打开 Dashboard 最近事件。
4. 确认能看到 inbound received、route resolved、agent turn completed、delivery enqueued、delivery sent。
5. 如果失败，查看最近错误，定位是飞书入站、路由、模型、工具、投递还是通道发送问题。
```

完整链路结论不要只看一个 QPS。应该拆成：

- `model-real` 的端到端模型耗时。
- `feishu-send-real` 的真实飞书发送耗时。
- Dashboard 事件流中真实飞书入站到投递完成的链路耗时。

后续如果要自动化完整链路压测，需要新增 `feishu-webhook-real` 场景，自动构造飞书 Webhook 事件并等待最终投递结果。

### 4.7 故障注入

最后做故障注入，确认系统是“报错可定位”还是“直接崩掉”。

- 停掉 RabbitMQ，观察投递是否明确失败。
- 停掉 PostgreSQL，观察状态写入是否受影响。
- 停掉 Redis，观察去重和限流是否退化但不阻断主链路。
- 故意填错模型 `API key` 或 `base_url`，观察错误分类是否准确。
- 故意填错飞书验签或加密配置，观察入站是否被正确拒绝。

## 5. 怎么判断边界

当出现以下任一情况时，通常说明已经接近边界：

- 并发再加大，吞吐不再增长。
- `P95` 比上一档明显翻倍。
- 错误率开始上升。
- 队列积压持续增长且不回落。
- 外部 API 限流或超时开始集中出现。

## 6. 建议记录项

每次压测都建议记录下面这些内容：

- 压测日期和机器配置。
- Git commit。
- 场景名。
- 请求数和并发数。
- 成功数和失败数。
- 吞吐量和 `QPS`。
- `E2E P50 / P95 / P99`。
- `Agent P50 / P95 / P99`。
- `Delivery P50 / P95 / P99`。
- 最大队列积压。
- 主要瓶颈判断。

## 7. 报告产物

建议保留下面两类文件：

- `workspace/reports/load-tests/*.json`
- `workspace/reports/capacity-baseline.md`

前者保留原始压测结果，后者用于汇总不同场景的容量边界和瓶颈判断。

生成汇总报告：

```bash
python scripts/build_capacity_baseline.py
```

## 8. 结论口径

最终报告建议回答四个问题：

1. 系统在当前机器上大约能跑到多少 `QPS`。
2. 哪个场景最先成为瓶颈。
3. 从哪个并发点开始出现明显退化。
4. 故障发生时是否能保持可定位、可恢复、可重试。

建议输出示例：

```text
Gateway 自身调度 QPS：来自 mock-local，不包含真实模型和飞书。
可靠投递 QPS：来自 delivery-rabbitmq，代表 RabbitMQ + DeliveryRuntime 的出站处理能力。
真实模型端到端 QPS：来自 model-real，主要受模型 API 和上下文长度影响。
真实飞书发送 QPS：来自 feishu-send-real，主要受飞书接口延迟和限流影响。
当前主要瓶颈：根据 P95、错误率和队列积压判断。
```
