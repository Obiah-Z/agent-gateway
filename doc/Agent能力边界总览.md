# Agent 能力边界总览

本文档记录 Gateway 当前 Agent 分层、主要职责和工具边界，用于后续继续扩展能力时避免职责重叠。

## 平台入口 Agent

`main`、`feishu-entry` 和 `wework-entry` 是入口层。入口层负责普通问答、意图分类、标准交接提示和委派建议，不负责深度执行。

关键工具包括 `prepare_entry_route_response`、`classify_task_intent`、`build_agent_handoff_prompt`、`compose_agent_handoff_package`、`format_agent_handoff_package`、`build_collaboration_stage_handoff`、`summarize_collaboration_progress`、`format_collaboration_progress`、`compose_collaboration_final_summary`、`format_collaboration_final_summary`、`list_agent_capabilities`、`format_agent_capability_catalog`、`match_agent_capability`、`format_agent_capability_match`、`explain_agent_capability_contract`、`format_agent_capability_contract`、`list_agent_collaboration_routes`、`plan_agent_collaboration`、`explain_agent_route`、`format_entry_response` 和 `suggest_agent_delegation`。其中 `prepare_entry_route_response` 只把分类、协作路线、能力目录、推荐结果、交接包、路由解释和用户回复组合成入口层准备结果，不执行目标 Agent；`suggest_agent_delegation` 只表示建议，不代表目标 Agent 已经自动执行；`compose_agent_handoff_package` 只把推荐结果转换成 handoff_prompt 和结构化委派建议，`format_agent_handoff_package` 只把交接包格式化成用户可读说明；`list_agent_capabilities` 读取当前配置和提示词形成真实能力目录，并标出 entry、shared-capability、personal、ops 等分层；`format_agent_capability_catalog` 只把目录格式化成用户可读说明，`match_agent_capability` 只基于目录为用户目标推荐 Agent，`format_agent_capability_match` 只把推荐结果格式化成用户可读说明；`explain_agent_capability_contract` 只解释某类任务的目标 Agent、关键工具、只读/写入、确认要求和协作模式，`format_agent_capability_contract` 只把该契约转成用户可读中文说明，二者都不执行目标 Agent 或写入数据；`list_agent_collaboration_routes` 只列出可用协作路线、别名和阶段模板；`plan_agent_collaboration` 只生成多 Agent 协作路线，不自动调用任何 Agent；`summarize_collaboration_progress` 只根据阶段输出判断协作进度和下一阶段，`format_collaboration_progress` 只把该进度 JSON 转成用户可读说明；`compose_collaboration_final_summary` 只把已完成阶段结果收束为稳定的 `agent_collaboration_final_summary` JSON，`format_collaboration_final_summary` 只把该 JSON 转成用户可读最终摘要，二者都不重新执行任何 Agent；`build_collaboration_stage_handoff` 只把某一阶段和上游结果整理成可复制交接提示；`explain_agent_route` 只解释为什么选择某个 Agent 或协作路线，不代表目标 Agent 已经执行。

复杂 GitHub 仓库任务如果同时要求分析、风险、采纳计划或正式报告，入口层应使用 `repo-adoption` 协作路线：repo-analyzer 先产出仓库分析和风险扫描，reviewer 做仓库风险门禁，planner 整合成采纳执行计划，doc-writer 最后成文落盘。

复杂技术选型、方案对比或中间件取舍任务如果同时要求验证计划、风险审查、落地计划或正式报告，入口层应使用 `research-option-validation` 协作路线：research 先产出方案对比，reviewer 做方案门禁，planner 生成最小验证计划，reviewer 审查计划门禁，doc-writer 最后成文落盘。

`classify_task_intent` 会把普通仓库理解类请求归为 `repo-analysis`，把包含风险、采纳、引入、复用、计划或报告要求的复杂仓库请求归为 `repo-adoption`，并返回 `requires_collaboration=true` 与 `collaboration_task_type=repo-adoption`；也会把包含技术选型、方案对比或中间件取舍，同时要求验证计划、风险审查、落地计划或正式报告的请求归为 `research-option-validation`，供入口层触发协作路线。用户询问当前有哪些 Agent、每个 Agent 能做什么或任务该交给谁时，归为 `agent-capabilities`，由入口层调用 `list_agent_capabilities` 和 `format_agent_capability_catalog` 直接回答。

入口层生成 `agent_collaboration_plan` 后，应把该 JSON 作为 `collaboration_plan_json` 传给 `format_entry_response`，向用户展示阶段路线和“尚未自动执行”的边界，而不是只输出单个 Agent 的委派建议。

## 共享能力 Agent

`research` 负责联网检索、来源核验、证据包沉淀和方案选型对比。`assess_research_confidence` 用于评估来源质量、结论置信度、冲突和验证缺口，`format_research_confidence_assessment` 用于把该评估转成用户可读中文报告；`compose_research_brief` 用于把调研结论、来源、不确定点和可复用摘要整理成结构化简报，`format_research_brief` 用于把该简报转成用户可读中文摘要；`compose_research_evidence_pack` 用于把来源、关键事实、不确定点和下游用途整理成可复用材料；`compose_research_option_comparison` 用于把技术选型、方案对比或中间件取舍整理成候选方案、评价维度、推荐项、来源依据和下游动作。

`repo-analyzer` 负责 GitHub 仓库分析、Gateway 适配评估、轻量风险扫描、快速决策卡片、阅读路线和采纳路线图。`github_repo_decision_card` 用于回答“值不值得看、是否继续深入、下一步先看什么”这类轻量判断，`format_github_repo_decision_card` 用于把该结构化判断转成用户可读中文摘要；`github_repo_reading_guide` 用于回答“这个仓库先看哪些文件、从哪里读起、阅读顺序是什么”，`format_github_repo_reading_guide` 用于把阅读路线转成用户可读中文摘要；`compose_github_repo_analysis` 用于把仓库定位、适配评估、关键发现、风险和建议固化为结构化分析，`format_github_repo_analysis` 用于把该分析转成即时中文摘要；`plan_github_repo_adoption` 用于把结构化仓库分析转成采纳决策、阶段路线、风险门槛和验收项，`format_github_repo_adoption_plan` 用于把采纳路线图转成用户可读中文摘要；`github_repo_risk_scan` 用于检查许可证、维护状态、README 证据、issue 数和依赖文件信号；repo-analyzer 不负责正式文档成文和任务执行。

`planner` 负责阶段计划、执行拆解、采纳计划转换、仓库审查结果整合、方案验证计划和协作路线转换。`format_task_breakdown` 和 `format_execution_stage_plan` 用于把 planner 的结构化 JSON 转成用户可读摘要，避免直接暴露机器传递格式；`compose_repo_review_task_plan` 用于把 repo-analyzer 的 `github_repo_analysis`、reviewer 的 `github_repo_risk_gate_review` 和可选 `github_repo_adoption_plan` 合并为可落盘的仓库采纳执行计划；`compose_research_option_validation_plan` 用于把 research 的 `research_option_comparison` 和 reviewer 的 `research_option_comparison_gate_review` 转成最小验证计划，no-go 时只能安排补证，不能直接进入实现；`adapt_collaboration_plan_to_task_plan` 用于把入口 Agent 的 `agent_collaboration_plan` 转成可落盘的阶段计划，明确每一阶段交给哪个 Agent、输入依据、输出和完成标准，但不自动调用任何 Agent。

`reviewer` 负责风险审查、发布门禁、计划门禁、协作路线门禁、交接包门禁、协作进度门禁、协作最终摘要门禁、证据复用门禁、方案对比门禁和仓库风险门禁。`assess_risk_decision` 用于把审查发现、测试缺口、残余风险和证据等级转换为稳定风险分、推进判断和优先动作，`format_risk_decision_assessment` 用于把风险评分结构化结果转成用户可读审查摘要；`review_release_gate` 用于检查发布、合并或推送前是否具备变更摘要、测试证据、无阻塞未决项和回滚方案，`format_release_gate_review` 用于把发布门禁结构化结果转成用户可读审查摘要；`review_research_evidence_gate` 用于检查 research 证据包是否具备问题、结论、来源 URL、一手来源、关键事实、不确定点和时效说明，`format_research_evidence_gate_review` 用于把证据门禁结构化结果转成用户可读审查摘要；`review_research_option_comparison_gate` 用于检查 research 方案对比是否具备决策问题、候选方案、评价维度、来源、一手来源、推荐项、候选方案证据和不确定点，避免未经审查的选型材料直接进入 planner 或 doc-writer，`format_research_option_comparison_gate_review` 用于把方案对比门禁结构化结果转成用户可读审查摘要；`review_task_plan_gate` 用于检查阶段计划、采纳计划和方案验证计划是否具备目标、边界、阶段、完成标准、风险和验收依据，输入是 `task_plan_from_research_option_comparison` 时还会检查方案门禁、推荐方案、候选方案、评价维度和执行动作限制；`format_task_plan_gate_review` 用于把计划门禁结构化结果转成用户可读审查摘要；`review_agent_collaboration_gate` 用于检查入口 Agent 生成的多 Agent 协作路线是否具备目标、交接契约、阶段输出、约束和未自动执行声明；`format_agent_collaboration_gate_review` 用于把协作路线门禁结构化结果转成用户可读审查摘要；`review_agent_handoff_package_gate` 用于检查入口 Agent 生成的 `agent_handoff_package` 是否具备目标、用户目标、提示结构、约束边界、推荐依据和未自动执行声明；`format_agent_handoff_package_gate_review` 用于把交接包门禁结构化结果转成用户可读审查摘要；`review_collaboration_progress_gate` 用于检查入口 Agent 生成的协作进度是否具备连续阶段、下一阶段、handoff 参数、上游结果和风险边界；`format_collaboration_progress_gate_review` 用于把协作进度门禁结构化结果转成用户可读审查摘要；`review_collaboration_final_summary_gate` 用于检查入口 Agent 生成的 `agent_collaboration_final_summary` 是否具备明确最终结论、完整阶段覆盖、完成状态一致性、阶段依据、下一步动作和未重新执行声明；`format_collaboration_final_summary_gate_review` 用于把协作最终摘要门禁结构化结果转成用户可读审查摘要；`review_github_repo_risk_gate` 用于检查 repo-analyzer 的 `github_repo_risk_scan` 是否具备明确用途、许可证判断、维护状态、高危阻塞风险和缓解动作，`format_github_repo_risk_gate_review` 用于把仓库风险门禁结构化结果转成用户可读审查摘要。reviewer 不直接修改系统或执行高风险动作。

`doc-writer` 负责正式 Markdown 成文。`render_repo_analysis_markdown` 用于把 repo-analyzer 的仓库分析结果渲染为正式报告，`render_github_repo_risk_markdown` 用于把 `github_repo_risk_scan` 渲染为仓库风险扫描文档，并可合并 reviewer 的 `github_repo_risk_gate_review` 门禁结论；`render_research_evidence_markdown` 用于把 research 的证据包渲染为调研证据文档，`render_research_option_comparison_markdown` 用于把 research 的 `research_option_comparison` 渲染为方案对比或技术选型文档，`render_research_option_validation_plan_markdown` 用于把 planner 的 `task_plan_from_research_option_comparison` 渲染为方案验证计划，`render_execution_record_markdown` 用于把 planner/reviewer 的结构化结果渲染为执行记录，`render_release_gate_markdown` 用于把 reviewer 的 `release_gate_review` 渲染为正式发布门禁审查报告，`render_agent_collaboration_markdown` 用于把入口 Agent 生成的多 Agent 协作路线渲染为正式方案，`render_agent_collaboration_progress_markdown` 用于把入口 Agent 生成的协作进度渲染为正式进度文档，`render_collaboration_progress_gate_markdown` 用于把 reviewer 的 `collaboration_progress_gate_review` 渲染为正式门禁审查报告，并可合并原始 `agent_collaboration_progress`，`render_agent_handoff_package_gate_markdown` 用于把 reviewer 的 `agent_handoff_package_gate_review` 渲染为正式交接包门禁审查报告，并可合并原始 `agent_handoff_package`，`render_agent_collaboration_final_summary_markdown` 用于把入口 Agent 的 `agent_collaboration_final_summary` 渲染为正式最终摘要，`render_collaboration_final_summary_gate_markdown` 用于把 reviewer 的 `collaboration_final_summary_gate_review` 渲染为正式最终摘要门禁审查报告，并可合并原始 `agent_collaboration_final_summary`。它只表达路线、进度、门禁结论、阶段结果和交接契约，不代表任何 Agent 已经自动执行。

## 个人 Agent

`personal-secretary-zhanghaibo` 只服务指定企业微信用户，负责待办、复盘、个人简报、时间块、每日工作流、当前聚焦卡片、个人日复盘草稿、周计划草稿和收件箱整理闭环。`personal_todo_add` 用于写入结构化待办，`format_personal_todo_entry` 用于把新增待办结果转成用户可读中文确认；`personal_todo_list` 用于查询结构化待办，`personal_todo_search` 用于按标题、备注、结果或取消原因搜索结构化待办，`format_personal_todo_list` 用于把待办列表或搜索候选转成用户可读中文摘要；`personal_todo_status_card_generate` 用于生成未完成、高优先级、最近完成和最近取消的只读待办状态卡，`format_personal_todo_status_card` 用于把状态卡转成用户可读中文摘要；`personal_due_todo_digest_generate` 用于按 due_at 汇总已到期、今天、明天、本周和下周的只读待办提醒摘要，`format_personal_due_todo_digest` 用于把提醒摘要转成用户可读中文摘要；`personal_todo_complete` 用于按 todo_id 完成指定待办，`personal_todo_complete_by_title` 用于按标题片段唯一匹配并完成待办，多匹配时不自动完成，`personal_todo_update_by_title` 用于按标题片段唯一匹配并更新待办标题、优先级、时间或备注，多匹配时不自动修改，`personal_todo_cancel_by_title` 用于按标题片段唯一匹配并取消待办，多匹配时不自动取消，`personal_todo_reopen_by_title` 用于按标题片段唯一匹配并恢复已完成或已取消待办，多匹配时不自动恢复，`format_personal_todo_update`、`format_personal_todo_cancellation`、`format_personal_todo_reopen` 和 `format_personal_todo_completion` 用于把更新、取消、恢复或完成结果转成用户可读中文确认；`personal_review_add` 用于写入结构化复盘，`format_personal_review_entry` 用于把新增复盘结果转成用户可读中文确认；`personal_review_recent` 用于查询近期结构化复盘，`personal_review_search` 用于按摘要、完成事项、卡点或下一步关键词搜索历史复盘，`format_personal_review_recent` 用于把复盘列表、搜索结果、卡点和下一步线索转成用户可读中文摘要；`personal_briefing_generate` 用于汇总未完成待办、紧急项、近期复盘和建议焦点，`format_personal_briefing` 用于把个人简报转成用户可读中文摘要；`personal_time_blocks_generate` 用于把未完成待办按上午、下午、晚上生成时间块，`format_personal_time_blocks` 用于把时间块计划转成用户可读中文摘要；`personal_daily_workflow_generate` 用于把待办、近期复盘、时间块、第一步和确认项组合成结构化每日工作流，`format_personal_daily_workflow` 用于把该工作流转成用户可读中文摘要；`personal_focus_card_generate` 用于把未完成待办和近期复盘收敛成一个当前最该推进的动作，`format_personal_focus_card` 用于把聚焦卡片转成用户可读中文摘要；`personal_day_review_plan_generate` 用于生成日复盘和明日计划草稿，`format_personal_day_review_plan` 用于把草稿转成用户可读中文摘要；`personal_weekly_plan_generate` 用于生成周计划草稿，`format_personal_weekly_plan` 用于把周计划转成用户可读中文摘要；`personal_inbox_triage` 用于把混杂输入整理成待办、复盘和长期记忆候选，`format_personal_inbox_triage` 用于把整理结果转成用户可读中文摘要；`personal_inbox_commit` 只在用户确认后批量写入明确待办和复盘，不自动写入长期记忆，`format_personal_inbox_commit` 用于把批量写入结果转成用户可读中文确认；`format_memory_write` 用于把已确认的长期记忆写入结果转成用户可读中文确认，不负责自动判断或扩大记忆写入范围；`format_memory_search` 用于把长期记忆检索结果转成用户可读中文摘要，只读不写。

`diet-assistant-zhanghaibo` 只服务指定企业微信用户，负责饮食档案、餐食、体重、饮食计划、今日营养汇总、进展统计、趋势简报、今日饮食闭环、下一餐建议卡片、饮食日总结草稿、周饮食计划草稿和混合饮食输入整理。`profile_get` 用于查询已保存饮食档案，`format_diet_profile` 用于把档案、偏好限制和缺失字段转成用户可读中文摘要；`profile_update` 用于写入身高、目标、偏好、忌口等饮食档案字段，`format_diet_profile_update` 用于把档案更新结果转成用户可读中文确认；`meal_log_add` 用于写入单餐记录，`format_meal_log_entry` 用于把刚保存的餐食、热量和三大营养素转成用户可读中文确认；`meal_log_update` 用于按 meal_id 修正已保存餐食的内容、日期、餐次、热量和三大营养素，`format_meal_log_update` 用于把修正结果转成用户可读中文确认；`weight_log_add` 用于写入体重记录并更新当前体重档案，`format_weight_log_entry` 用于把体重记录结果转成用户可读中文确认；`weight_log_list` 用于查询最近体重记录，`format_weight_log_list` 用于把体重明细和轻量趋势转成用户可读中文摘要；`weight_log_update_latest` 用于修正最近一条体重记录并同步当前体重档案，`format_weight_log_update` 用于把修正结果转成用户可读中文确认；`diet_inbox_triage` 用于把用户一次性输入的餐食、体重、目标和偏好拆成候选记录与确认项，`format_diet_inbox_triage` 用于把整理结果转成用户可读中文摘要，`diet_inbox_commit` 只在用户确认后批量写入明确餐食、体重和安全档案字段，不自动写入长期偏好或记忆，`format_diet_inbox_commit` 用于把批量写入结果转成用户可读中文确认；`format_memory_write` 用于把已确认的长期饮食偏好记忆写入结果转成用户可读中文确认，不负责自动写入记忆；`format_memory_search` 用于把长期饮食偏好检索结果转成用户可读中文摘要，只读不写；`diet_today_status` 用于读取当天餐食、体重、计划、趋势和风险，`format_diet_today_status` 用于把该状态转成用户可读中文状态卡，二者不自动写入记录或生成计划；`meal_log_list` 用于查询已保存餐食记录，`format_meal_log_list` 用于把餐食记录列表转成用户可读中文摘要；`nutrition_day_summary` 用于生成并保存当天营养汇总，`format_nutrition_day_summary` 用于把今日营养汇总转成用户可读中文摘要；`diet_plan_generate` 用于生成并保存当天饮食计划，`format_diet_plan` 用于把当天计划转成用户可读中文摘要；`progress_summary` 用于读取近期餐食、每日统计和体重变化，`format_diet_progress_summary` 用于把进展统计转成用户可读中文摘要；`diet_coach_briefing` 用于读取近期餐食和体重趋势生成教练简报，`format_diet_coach_briefing` 用于把趋势简报转成用户可读中文摘要；`diet_daily_loop_generate` 用于读取今日餐食、计划、体重、风险和下一步动作，`format_diet_daily_loop` 用于把今日闭环转成用户可读中文摘要；`diet_next_meal_card_generate` 用于根据今日餐食、计划、热量缺口和风险生成只读下一餐建议，`format_diet_next_meal_card` 用于把该建议卡片转成用户可读中文摘要；`diet_day_review_plan_generate` 用于生成饮食日总结和明日建议草稿，`format_diet_day_review_plan` 用于把草稿转成用户可读中文摘要；`diet_weekly_plan_generate` 用于生成饮食周计划草稿，`format_diet_weekly_plan` 用于把周计划转成用户可读中文摘要。上述查询、计划和整理类工具只读取已有记录或候选输入生成建议，不自动补记餐食、写体重或生成新计划；餐食修正只更新已匹配 meal_id，不新增重复餐食。

## 运维 Agent

`ops` 负责只读运维诊断。`ops_readonly_health` 采集基础健康信息，`summarize_ops_health` 生成巡检摘要，`ops_runtime_diagnostics` 汇总运行事件、失败投递和告警线索，`ops_troubleshooting_plan` 把健康摘要和运行诊断整理成只读排障行动清单。ops 不执行删除、清空、重启、改权限或改配置。

## 扩展原则

新增能力时优先判断应该落在哪一层：入口层只做分类和交接，共享能力层做通用专业能力，个人 Agent 做用户私有闭环，ops 做只读排障。

新增工具应同时更新 `config/agents.json`、对应 `workspace/agents/*` 提示词和测试。写入类工具必须明确数据范围和用户作用域；草稿类工具必须明确不会自动写入。
