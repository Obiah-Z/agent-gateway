# 入站任务分发演进复盘

本文用于复盘 Gateway 入站任务分发从单进程串行执行演进到分布式会话调度的过程。重点不是只介绍最终方案，而是说明为什么会一步步走到现在的设计，以及每个阶段遇到的问题、技术选型和取舍。

这部分可以作为面试时回答“项目最大的难点是什么、你是怎么解决的”的素材。

## 1. 一句话总结

这个项目入站任务分发最大的难点，是从单进程执行演进到多 worker 分布式执行后，原本在单机里自然成立的顺序和互斥约束都消失了。

我最终把系统拆成了四层：

```text
PostgreSQL / TaskStore 负责事实状态
Redis 负责 session 级调度和互斥
RabbitMQ 负责削峰和唤醒
Worker 池负责真正执行 Agent 任务
```

最终效果是：

```text
同一 session 严格串行
不同 session 并行执行
热点 session 不阻塞其他 session
RabbitMQ 不再决定具体执行顺序
Redis 调度索引可观测、可重建
```

## 2. 第一阶段：单进程串行执行

项目最早期的模型比较直接。飞书、CLI 或其他通道消息进入系统后，在同一个进程里完成路由、模型调用、工具调用和回复发送。

早期链路大致是：

```text
外部消息
  -> ChannelRuntime
  -> GatewayDispatcher
  -> AgentLoopRunner
  -> Tool Calling
  -> Delivery
```

这个阶段的优点是简单：

- 所有逻辑都在一个进程里，状态共享容易。
- 同一个用户连续发消息时，通常天然按顺序处理。
- 不需要考虑跨 worker 的锁、续租和接管。
- 出问题时看日志就能定位。

但是随着系统能力变多，问题开始暴露：

- 模型调用和工具调用可能很慢，入口线程容易被长任务拖住。
- 飞书 Webhook 需要快速 ACK，不能一直等模型执行完成。
- Cron、Heartbeat、Skill、GitHub 仓库分析等任务都可能是长任务。
- 单进程无法横向扩展，也缺少任务失败后的重试和恢复能力。

所以第一步改造目标很明确：把“入口接收”和“任务执行”拆开。

## 3. 第二阶段：引入后台任务队列

这一阶段把非 CLI 入站消息转成 `agent_inbound` 后台任务。入口只负责解析、验签、去重、标准化和创建任务，真正的 Agent 执行交给后台 worker。

链路变成：

```text
外部消息
  -> ChannelRuntime
  -> TaskQueue.enqueue()
  -> TaskStore 创建 agent_inbound task
  -> TaskWorkerRuntime
  -> GatewayDispatcher
  -> AgentLoopRunner
```

这个改造解决了入口阻塞问题。Webhook 可以快速返回，长任务可以在后台慢慢跑。

但它也带来了新的问题：一旦任务可以被多个 worker 消费，同一个 session 的多条消息就可能被并发执行。

例如用户连续发送：

```text
A1 -> A2 -> A3
```

如果有多个 worker，同时拿到 A1 和 A2，就会出现：

```text
worker-1 执行 A1
worker-2 执行 A2
```

这在普通后台任务系统里可能可以接受，但对 Agent 会话不行。因为 Agent 依赖上下文历史，同一 session 并发执行会导致：

- 会话历史写入交错。
- 模型上下文读取到旧状态。
- 工具调用结果顺序错乱。
- 重复回复或回复顺序颠倒。
- 记忆写入污染。

所以进入分布式执行后，第一个必须解决的问题是“互斥执行”。

## 4. 第三阶段：用 Redis 实现 session 互斥

互斥的目标是：

```text
同一个 session 同一时间只能有一个 worker 执行
```

我最先考虑的是 Redis 分布式锁，因为它适合做高频、短生命周期的协调状态。

核心做法是：

```text
key   = gateway:lock:agent_inbound:{session_key}
value = worker_id + task_id + owner metadata
ttl   = lock_ttl_seconds
```

执行前：

```text
SET key value NX EX ttl
```

释放时：

```text
Lua 校验 value 匹配后 DEL
```

续租时：

```text
Lua 校验 value 匹配后 EXPIRE
```

这个阶段引入了 `RedisLaneCoordinator`，把普通 Redis 锁升级成 session lane ownership。它不只是一个锁，还包含 owner metadata、续租、释放、stale owner 观测和状态镜像。

这个阶段解决了最危险的问题：同 session 不会被多个 worker 同时执行。

但是它只解决“互斥”，没有彻底解决“顺序”。

## 5. 第四阶段：引入 RabbitMQ 做入站分发

为了支持多 worker 和削峰，系统进一步引入 RabbitMQ 作为入站任务 broker。

当任务创建后，系统把轻量任务引用发布到 RabbitMQ：

```json
{
  "task_id": "ed9c96a187c546c2",
  "task_type": "agent_inbound",
  "session_key": "agent:main:feishu:feishu-main:direct:ou_xxx",
  "partition": 3,
  "idempotency_key": "..."
}
```

RabbitMQ 不保存完整用户消息，只保存 task_id 等轻量引用。真正的任务内容仍在 TaskStore / PostgreSQL 中。

这一阶段的链路是：

```text
TaskQueue.enqueue()
  -> TaskStore / PostgreSQL 创建任务
  -> RabbitMQ 发布 task_id
  -> worker 消费 task_id
  -> TaskStore.reserve_task_id()
  -> RedisLaneCoordinator 获取 session lock
  -> Agent 执行
```

这一步的收益是：

- 入站任务可以削峰。
- worker 可以横向扩展。
- RabbitMQ 提供 ACK / NACK / DLQ。
- broker 消息只携带轻量引用，任务事实状态可恢复。

但这时又出现一个新问题：如果希望 RabbitMQ 本身提供顺序保障，通常要把 prefetch 设置得很保守，甚至设置为 1。

## 6. 第五阶段：RabbitMQ prefetch=1 的局限

最直观的想法是：同一个 session 的消息进入同一个 RabbitMQ 分区队列，然后把 `prefetch=1`，让 worker 一次只拿一条消息。

这样看起来可以保证：

```text
同一个分区内按队列顺序消费
```

但它的问题在于，RabbitMQ 的调度粒度是“队列 / 分区”，不是“session”。

例如一个分区里有多个 session：

```text
A1 -> A2 -> B1 -> C1
```

如果 A1 正在执行，A2 排在队头。此时 A2 因为 session A busy 不能执行，但 B1 和 C1 其实是可以执行的。

如果严格依赖 RabbitMQ FIFO，就会出现：

```text
A2 挡在队头
B1 / C1 被 A2 间接阻塞
```

这就是典型的队头阻塞。

换句话说，`prefetch=1` 虽然降低了乱序风险，但牺牲了吞吐和公平性。热点 session 会拖慢同一个分区里的其他 session。

如果把 prefetch 调大，又会遇到另一个问题：

- 同 session 的 A2/A3 可能被提前预取。
- worker 发现 session busy 后只能 retry / requeue。
- 顺序可能受到重试时间和重新投递影响。
- worker 资源会被不可执行任务消耗。

所以这个阶段得出的结论是：

```text
RabbitMQ 适合做可靠分发和削峰，但不适合承担最终的 session 调度决策。
```

## 7. 关键认知转变：队列不等于调度器

这个项目里最关键的一次认知转变是：不能把 RabbitMQ 队列当作完整调度器。

RabbitMQ 擅长的是：

- 消息可靠排队。
- ACK / NACK。
- 死信。
- 削峰。
- 唤醒消费者。

但它不知道：

- 哪个 session 当前正在执行。
- 哪个 session 当前可执行。
- 哪个任务应该因为 session busy 暂时等待。
- 哪些其他 session 可以绕过热点 session 继续执行。

所以要把职责拆开：

```text
RabbitMQ 管“有任务来了”
Redis 管“哪个 session 能执行”
PostgreSQL 管“任务事实状态”
Worker 管“真正执行任务”
```

这就是最终方案的基础。

## 8. 第六阶段：Redis ready index + session pending bucket

最终方案引入 `RedisSessionReadyScheduler`。

它使用三个 Redis key：

```text
gateway:tasks:sessions:ready
gateway:tasks:session:{session_key}:pending
gateway:tasks:session:{session_key}:busy
```

含义是：

| Key | 作用 |
| --- | --- |
| `gateway:tasks:sessions:ready` | 全局 ready index，只保存当前可以尝试执行的 session_key。 |
| `gateway:tasks:session:{session_key}:pending` | 某个 session 自己的待执行任务队列。 |
| `gateway:tasks:session:{session_key}:busy` | 当前 session 的执行 owner，带 TTL 和续租信息。 |

新的入站链路变成：

```text
入站消息
  -> TaskStore / PostgreSQL 先写任务事实状态
  -> Redis pending bucket 写入 task_id|task_type
  -> Redis ready index 写入 session_key
  -> RabbitMQ 发布 task_id 唤醒 worker
```

worker 执行时不再直接相信 RabbitMQ payload 中的 task_id，而是：

```text
1. 收到 RabbitMQ 唤醒消息
2. 去 Redis ready index claim 一个可执行 session
3. 从该 session 的 pending bucket 弹出队首 task
4. 写入 busy owner
5. 回到 TaskStore.reserve_task_id(task_id) 做事实状态抢占
6. 执行 Agent
7. 完成后 release busy owner
8. 如果 pending bucket 还有任务，把 session 放回 ready index
```

核心变化是：

```text
RabbitMQ 消息从“执行命令”变成“唤醒信号”
Redis scheduler 才决定真正执行哪个 session 的哪个队首任务
```

## 9. 为什么 ready index 只放 session_key

ready index 不放具体任务 ID，而只放 session_key，这是这个方案的关键。

如果 ready index 直接放任务：

```text
ready index = [A1, A2, A3, B1, C1]
```

当 A1 正在执行时，A2/A3 仍然会在全局队列里反复被 worker 拿到，然后发现 session A busy，再 requeue 或 retry。

这会导致：

- worker 资源被不可执行任务浪费。
- A2/A3 影响 B1/C1。
- retry 后顺序更难控制。

现在的设计是：

```text
ready index = [session-A, session-B, session-C]

session-A pending bucket = [A1, A2, A3]
session-B pending bucket = [B1]
session-C pending bucket = [C1]
```

worker claim 到 `session-A` 后，只弹出 A1，并把 session-A 标记 busy：

```text
session-A pending bucket = [A2, A3]
session-A busy = A1
ready index = [session-B, session-C]
```

这样 A2/A3 不会进入全局 ready index，不会堵住 B/C。等 A1 完成后，如果 A 的 bucket 还有任务，再把 session-A 放回 ready index。

这就实现了：

```text
热点 session 只阻塞自己，不阻塞其他 session
```

## 10. Claim / Release 的原子性

因为是分布式 worker，claim 和 release 必须是原子的。

`claim_next()` 用 Lua 脚本完成：

```text
1. 从 ready index 弹出 session_key
2. 检查 busy key 是否存在
3. 如果不 busy，从该 session pending bucket 弹出队首 task
4. 写入 busy owner，设置 TTL
5. 返回 task_id / session_key / owner_value
```

`release()` 也用 Lua 脚本完成：

```text
1. 校验 owner_value 是否匹配
2. 匹配才删除 busy key
3. 如果 pending bucket 还有任务，把 session_key 放回 ready index
4. owner 不匹配则拒绝释放
```

owner 校验很重要。它可以避免旧 worker 卡顿恢复后，误删新 worker 的 busy owner。

## 11. 长模型调用与 Watchdog 续租

Agent 任务和普通 Web 后端任务不同，它经常会卡在模型调用、工具调用或外部 API 上。

如果 busy owner TTL 太短，任务还没执行完，锁先过期，后续同 session 任务就可能被其他 worker 抢到。

所以当前方案里，worker 执行 scheduler claim 任务时，会启动 watchdog 续租：

```text
_execute_claimed_task()
  -> 启动 _renew_session_claim_until_cancelled()
  -> 执行 Agent 任务
  -> 取消续租协程
  -> release session claim
```

续租间隔默认是：

```text
min(60s, ttl / 3)
```

续租成功记录：

```text
task.scheduler.renewed
```

续租失败记录：

```text
task.scheduler.renew_failed
```

这里没有选择“续租失败就强行中断任务”，因为模型调用或工具调用可能已经产生外部副作用。更稳妥的方式是记录事件、暴露风险，依靠 TTL 和后续恢复机制处理。

## 12. PostgreSQL 为什么仍然是事实状态

Redis 只是调度索引，不保存完整任务事实。

真正的任务状态仍然在 PostgreSQL / TaskStore：

```text
task_id
task_type
source
status
agent_id
session_key
payload
retry_count
created_at
updated_at
```

这样设计有几个好处：

- Redis 丢数据后可以重建调度索引。
- RabbitMQ 清空后可以重新发布唤醒消息。
- Dashboard 可以查询任务生命周期。
- worker 执行前必须回到 TaskStore / PostgreSQL reserve，避免重复执行已完成任务。

当前重建入口是：

```text
tasks.scheduler.rebuild
```

它会扫描 `pending/retrying` 任务，重建：

```text
gateway:tasks:session:{session_key}:pending
gateway:tasks:sessions:ready
```

## 13. Prefetch 在最终方案里的角色

Phase 21 之后，RabbitMQ prefetch 不再直接影响同 session 的顺序，因为任务执行顺序由 Redis scheduler 决定。

但 prefetch 仍然会影响：

- 唤醒消息分发效率。
- worker 间公平性。
- 未 ACK 消息数量。
- worker 崩溃后的重新投递抖动。

所以不能说 prefetch 完全没影响。更准确的说法是：

```text
prefetch 不再决定会话顺序，但仍影响 RabbitMQ 唤醒消息的吞吐和公平性。
```

建议：

```text
GATEWAY_INBOUND_RABBITMQ_PREFETCH ≈ GATEWAY_TASK_WORKER_CONCURRENCY
```

例如：

```env
GATEWAY_INBOUND_RABBITMQ_PREFETCH=4
GATEWAY_TASK_WORKER_CONCURRENCY=4
```

## 14. 技术选型对比总结

| 阶段 | 方案 | 解决的问题 | 新暴露的问题 |
| --- | --- | --- | --- |
| 单进程串行 | 入口直接执行 Agent | 简单、天然顺序 | 入口阻塞、无法横向扩展、恢复能力弱 |
| 后台任务队列 | TaskStore + worker | 入口快速 ACK，长任务后台化 | 多 worker 下同 session 可能并发 |
| Redis 锁 | RedisLaneCoordinator | 解决同 session 互斥 | 不保证严格 FIFO，长任务要续租 |
| RabbitMQ 分区 | hash(session_key) -> partition | 削峰、分发、DLQ、降低乱序 | partition 粒度过粗，热点 session 队头阻塞 |
| RabbitMQ prefetch=1 | 保守消费 | 降低同分区乱序风险 | 吞吐低，其他 session 被热点 session 拖慢 |
| Redis session scheduler | ready index + pending bucket + busy owner | 同 session FIFO，不同 session 并行，热点只阻塞自己 | 实现复杂，需要观测、重建和续租 |

最终分层：

```text
PostgreSQL / TaskStore = 事实状态
Redis = 调度索引 + 执行互斥
RabbitMQ = 削峰 + 唤醒
Worker = 真正执行 Agent 任务
Dashboard / Control Plane = 观测 + 恢复
```

## 15. 面试表达版本

可以这样讲：

> 这个项目里我觉得最难的不是接一个模型 API，而是把 Agent 的入站任务从单进程执行改造成分布式执行后，仍然保证同一会话的上下文一致性。
>
> 一开始系统是单进程串行执行，天然不会乱序，但模型调用和工具调用很慢，Webhook 入口会被拖住，所以我先把入站消息抽象成后台任务，由 worker 异步执行。
>
> 这个改造之后出现了分布式场景的新问题：多个 worker 可能同时处理同一个 session 的消息，导致会话历史和上下文乱序。所以我先用 Redis 做 session 级互斥，通过 owner token、TTL 和续租保证同一 session 同一时间只有一个 worker 执行。
>
> 后来为了削峰和横向扩展，我引入 RabbitMQ 做入站任务分发，并按 session_key Hash 到不同分区。这个方案能提升分发能力，但我发现它的调度粒度还是分区，不是 session。如果把 prefetch 设为 1，虽然顺序更稳，但热点 session 会卡住同分区其他 session；如果 prefetch 调大，又会有同 session 后续任务被提前取出，出现同一 session 任务无法顺序消费的问题。
>
> 所以后面我把 RabbitMQ 的职责收敛为“削峰和唤醒”，不再让它决定具体执行哪个任务。真正的调度放到 Redis 里实现：全局 ready index 只放可运行的 session_key，每个 session 有自己的 pending bucket，busy owner 表示当前 session 正在执行。worker 被 RabbitMQ 唤醒后，去 Redis claim 一个可执行 session 的队首任务，再回到 PostgreSQL 抢占任务事实状态。
>
> 这样 A session 正在执行时，A2/A3 会留在 A 自己的 pending bucket，不会反复占用 worker，也不会堵住 B/C。A1 完成后，release 时发现 A bucket 还有任务，再把 A 放回 ready index。最终实现了同 session 严格 FIFO，不同 session 并行，热点 session 只阻塞自己。
>
> 同时我保留 PostgreSQL 作为任务事实状态，Redis 只是调度索引，RabbitMQ 只是唤醒队列。所以 Redis 索引丢了可以从 pending/retrying 任务重建，RabbitMQ 消息丢了也可以从事实状态重新发布。Dashboard 和控制面也能看到 ready session、pending bucket、busy owner，并支持手动 rebuild。

## 16. 面试追问要点

如果面试官追问“为什么不用 RabbitMQ 保证顺序”，可以回答：

> RabbitMQ 能保证队列维度的顺序，但我的业务约束是 session 维度的顺序。一个队列或一个 partition 里可能混有多个 session，所以 RabbitMQ FIFO 会造成热点 session 队头阻塞。最终我让 RabbitMQ 只做分发和唤醒，把 session 级调度放到 Redis。

如果追问“Redis pending bucket 能不能保证真实用户发送顺序”，可以回答：

> 它能保证 Redis 写入后的队列顺序和执行顺序，但如果多入口实例并发接收，同一 session 的平台事件到达 Redis 的顺序可能已经和用户真实发送顺序不同。要做到更强顺序，需要引入平台 event_time 或 session sequence。当前系统保证的是同 session 串行和按调度队列 FIFO 执行。

如果追问“worker 崩溃怎么办”，可以回答：

> busy owner 带 TTL，正常执行时 watchdog 续租。worker 崩溃后续租停止，TTL 到期后 session 可以恢复调度。任务事实状态仍在 PostgreSQL，Redis 调度索引也可以从 pending/retrying 任务重建。

如果追问“为什么 PostgreSQL 不是直接做调度器”，可以回答：

> PostgreSQL 更适合作为事实状态和恢复依据。Agent 任务执行时间长，如果把高频调度和长时间锁都压到数据库，会增加连接和锁管理成本。Redis 更适合热路径协调，PostgreSQL 保留强一致事实状态，两者职责分开。

| `agent_gateway/runtime/infra/rabbitmq.py` | RabbitMQ 入站 broker，负责 task_id 轻量引用和分区队列。 |
| `agent_gateway/runtime/execution/control_plane.py` | scheduler status / rebuild 控制面。 |
| `agent_gateway/monitoring/static/app.js` | Dashboard 展示 ready session、pending bucket、busy owner 和 rebuild 按钮。 |

## 18. 最终结论

这一路演进的核心不是简单“引入 Redis / RabbitMQ / PostgreSQL”，而是不断重新划分职责：

```text
单进程阶段：靠进程内顺序自然成立
后台任务阶段：解决入口阻塞
Redis 互斥阶段：解决同 session 并发
RabbitMQ 分发阶段：解决削峰和横向扩展
Redis scheduler 阶段：解决同 session 顺序和热点 session 队头阻塞
```

最终方案的关键价值在于：

```text
队列负责唤醒
Redis 负责调度
数据库负责事实
worker 负责执行
控制面负责观测和恢复
```
