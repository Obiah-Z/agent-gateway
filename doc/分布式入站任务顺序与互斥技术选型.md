# 分布式入站任务顺序与互斥技术选型

本文档用于复盘 Gateway 在引入入站任务队列后，如何在分布式 worker 场景下保证同一会话不并发执行，以及尽量保持同一会话消息顺序。

## 1. 背景

当前系统已经支持把非 CLI 入站消息从 `ChannelRuntime` 转入 `agent_inbound` 后台任务：

```text
飞书 / Telegram 入站
  -> ChannelRuntime
  -> agent_inbound task
  -> TaskWorkerRuntime
  -> GatewayDispatcher
  -> AgentLoopRunner
  -> DeliveryQueue
  -> RabbitMQ 出站通知
  -> DeliveryRuntime
  -> 外部通道出站
```

这个改造把慢模型调用从入站 lane 中移出，让入口更快释放，也让入站消息具备持久化任务状态。但在多 worker 或多实例部署后，会出现两个新问题：

- 同一 session 的多个 `agent_inbound` task 可能被不同 worker 并发执行。
- 同一 session 的消息可能因为重试、抢占或 worker 调度而乱序执行。

这会影响：

- 会话 JSONL / PostgreSQL transcript 写入顺序。
- 模型上下文重放顺序。
- 用户连续多轮对话的一致性。
- 工具调用结果和后续消息的对应关系。

## 2. 需要解决的问题

这里要区分两个目标：

| 目标 | 含义 | 重要性 |
| --- | --- | --- |
| 互斥执行 | 同一 session 同一时间只能有一个任务执行 | 必须解决 |
| 顺序执行 | 同一 session 的任务按入队顺序执行 | 强烈建议解决 |

互斥执行解决的是“并发写乱会话”的问题。

顺序执行解决的是“后一条消息先执行，导致上下文不符合用户真实顺序”的问题。

## 3. 方案一：Redis 分布式锁

### 设计

每个 `agent_inbound` task 执行前，基于 `session_key` 获取 Redis 锁：

```text
lock:agent_inbound:{session_key}
```

获取方式：

```text
SET lock_key worker_id NX EX 120
```

执行完成后，只有锁 value 仍然等于当前 worker_id 时才释放：

```text
if GET lock_key == worker_id:
    DEL lock_key
```

如果获取锁失败，任务不执行，重新进入 `retrying` 或延迟重试。

### 优点

- 实现成本低。
- Redis 天然适合跨进程、跨机器做轻量协调。
- 与 PostgreSQL、RabbitMQ、多个 worker 都能配合。
- 可以快速避免同一 session 并发执行。
- 锁过期后可以自动恢复，避免 worker 崩溃永久占锁。

### 缺点

- 主要解决互斥，不天然保证严格 FIFO。
- 需要处理锁续租，否则模型调用超过锁 TTL 时可能误并发。
- 需要防止误释放，必须使用 token/value 校验。
- Redis 不可用时要决定是拒绝、降级单机，还是冒险执行。
- 多 Redis 节点或 Redis Cluster 下还要考虑锁语义和故障恢复。

### 适用场景

- 当前阶段快速提高分布式 worker 安全性。
- 重点是防止同一 session 并发执行。
- 可以接受“近似顺序”，严格顺序留到后续阶段。

## 4. 方案二：PostgreSQL advisory lock

### 设计

执行 `agent_inbound` 前，基于 `session_key` 哈希获取 PostgreSQL advisory lock：

```sql
SELECT pg_try_advisory_lock(hashtext(:session_key));
```

执行完成后释放：

```sql
SELECT pg_advisory_unlock(hashtext(:session_key));
```

如果 worker 崩溃，数据库连接断开后锁会自动释放。

### 优点

- 与当前 PostgreSQL task 主存储贴合。
- 不需要引入 Redis 锁语义。
- 锁生命周期和数据库连接绑定，worker 崩溃后更容易自动释放。
- 对同一数据库内的多 worker 有明确互斥效果。

### 缺点

- 强依赖 PostgreSQL。
- 锁和业务执行耦合到数据库连接，长模型调用会长期占用连接或需要额外连接管理。
- 不天然保证严格 FIFO。
- 如果后续任务调度迁到 RabbitMQ，仍需要保留数据库锁或重新设计。
- advisory lock 的 hash 冲突虽然概率低，但需要注意 key 设计。

### 适用场景

- 单 PostgreSQL 主库是系统事实中心。
- worker 数量不大。
- 希望减少 Redis 分布式锁的复杂度。

## 5. 方案三：RabbitMQ 按 session 分区队列

### 设计

把 `agent_inbound` task 按 `session_key` 路由到固定分区：

```text
partition = hash(session_key) % N
routing_key = inbound.partition.{partition}
```

每个分区对应一条队列：

```text
agent_gateway.inbound.partition.0
agent_gateway.inbound.partition.1
...
agent_gateway.inbound.partition.N
```

同一个 session 永远进入同一条队列，队列内由 worker 串行消费。

### 优点

- 更接近严格顺序。
- 同一 session 天然落到同一条队列。
- 队列内 FIFO 能力比普通抢锁更强。
- 不需要每个任务都抢分布式锁。
- 横向扩展时可以按分区增加 worker。

### 缺点

- 架构改动较大。
- 分区数量需要提前规划。
- 热点 session 会拖慢所在分区。
- 扩缩容时 hash 分布变化会带来迁移问题。
- 需要设计失败重试、死信、重新投递和幂等消费。
- 如果同一队列多个 consumer 并发消费，仍要限制 prefetch 或 consumer 并发，否则顺序仍可能被破坏。

### 适用场景

- 系统已经明确进入多实例、多 worker 部署。
- 对同一 session 顺序要求较高。
- 愿意为更强顺序能力改造任务分发层。

## 6. 方案四：任务调度层 per-session lane

### 设计

在任务调度层维护类似 `ChannelRuntime lane` 的机制：

```text
session_key -> lane
lane 内串行
lane 间并发
```

区别是这个 lane 不再只存在于单进程内存，而是结合任务状态存储或分布式协调。

### 优点

- 和当前 ChannelRuntime lane 思路一致，容易解释。
- 既能控制同 session 串行，又能保持不同 session 并发。
- 可以统一管理优先级、取消、重试和长任务状态。
- 后续可以成为 Gateway 自己的任务调度核心。

### 缺点

- 实现复杂度最高。
- 需要设计 lane ownership、worker 心跳、超时接管和任务恢复。
- 多实例下必须依赖 Redis、PostgreSQL 或 RabbitMQ 协调 lane 归属。
- 需要处理热点 lane、worker 崩溃和迁移。

### 适用场景

- Gateway 进入长期生产化阶段。
- 希望自己掌控任务调度语义，而不是完全依赖 RabbitMQ 队列模型。
- 需要同时治理优先级、并发、顺序、取消和重试。

## 7. 对比总结

| 方案 | 互斥能力 | 顺序能力 | 实现成本 | 分布式适配 | 当前推荐度 |
| --- | --- | --- | --- | --- | --- |
| Redis 分布式锁 | 强 | 弱到中 | 低 | 强 | 高 |
| PostgreSQL advisory lock | 强 | 弱到中 | 中 | 中 | 中 |
| RabbitMQ 分区队列 | 中到强 | 强 | 高 | 强 | 中到高 |
| per-session lane | 强 | 强 | 最高 | 强 | 长期推荐 |

## 8. 推荐路线

### 短期：Redis 分布式锁

先解决最关键的问题：

```text
同一 session 不能被多个 worker 同时执行。
```

原因：

- 系统已经接入 Redis。
- 实现成本低。
- 可以快速保护会话写入顺序不被并发打乱。
- 不需要立即重构任务队列和 RabbitMQ 入站协议。

建议行为：

- `agent_inbound` 执行前获取 Redis 锁。
- 获取不到锁时，任务进入短延迟 retry。
- 锁 value 使用 worker_id + task_id，释放时校验 value。
- TTL 必须大于常见模型调用耗时。
- 长任务需要续租或设置较长 TTL。

### 中期：RabbitMQ session 分区

当入站吞吐和多 worker 数量继续增加后，引入 RabbitMQ 分区队列：

```text
同一 session -> 固定分区队列 -> 串行 worker
```

目标：

- 减少频繁抢锁。
- 让顺序能力更接近队列原生 FIFO。
- 为多实例入口和多 worker 消费提供更清晰的拓扑。

### 长期：per-session task lane

当系统需要更强任务治理时，把 per-session lane 提升为任务调度核心：

- 每个 session 有清晰 lane 状态。
- lane 可观察、可暂停、可迁移。
- 支持优先级、取消、重试、限流和热点治理。

## 9. 当前建议结论

当前阶段最合适的落地方案是：

```text
Redis 分布式锁 + PostgreSQL task 状态
```

它能用较低成本解决最危险的问题：多 worker 并发执行同一会话任务。

后续如果要进一步增强顺序性，再升级为：

```text
RabbitMQ session 分区队列
```

如果项目最终要长期运行并支持复杂调度，再演进到：

```text
分布式 per-session task lane
```

一句话总结：

```text
防并发：优先 Redis 锁。
保严格顺序：优先 RabbitMQ 分区或 per-session lane。
当前阶段：先 Redis 锁，后 RabbitMQ 分区，最终任务 lane 化。
```
