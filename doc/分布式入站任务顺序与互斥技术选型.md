# 分布式入站任务顺序与互斥技术选型

本文档用于说明 Gateway 在引入入站任务队列后，如何在多 worker、多实例场景下保证同一会话不被并发执行，并尽量避免热点会话拖慢其他会话。这里讨论的重点不是“是否使用 RabbitMQ”，而是要把消息分发、任务调度、会话互斥和状态持久化这几件事分清楚。

## 背景

Gateway 早期的入站链路相对直接，飞书、CLI 或其他通道的消息进入 `ChannelRuntime` 后，会继续经过路由、AgentLoop、工具调用、模型调用和出站投递。这个模型在单进程内比较容易理解，但当模型调用变慢、工具执行变长、飞书消息并发进入时，入口线程会被慢任务拖住，系统也缺少跨进程的恢复和调度能力。

因此系统开始把非 CLI 入站消息转成 `agent_inbound` 后台任务。新的链路大致是：

```text
飞书 / Telegram / Webhook
  -> ChannelRuntime
  -> agent_inbound task
  -> TaskWorkerRuntime
  -> GatewayDispatcher
  -> AgentLoopRunner
  -> DeliveryQueue
  -> RabbitMQ / 本地可靠投递
  -> 外部通道发送
```

这个改造把“入口接收”和“慢模型执行”拆开了，但也引入了新的分布式问题。多个 worker 同时消费任务时，同一 session 的多条消息可能被不同 worker 并发处理，导致会话历史、模型上下文、工具调用结果和记忆写入出现乱序。对 Agent Gateway 来说，同一会话的串行性是强约束，不同会话之间的并发性才是吞吐来源。

## 核心矛盾

这里要区分两个目标。第一个目标是互斥执行，即同一个 session 同一时间只能有一个任务进入 AgentLoop。这个目标必须满足，否则会直接破坏会话一致性。第二个目标是顺序执行，即同一个 session 的任务最好按用户真实发送顺序处理。顺序执行比互斥执行更难，因为它不仅要求同一时刻只有一个任务运行，还要求调度层不要让后来的任务抢在前面的任务之前执行。

单纯使用一个 FIFO 队列无法同时解决所有问题。例如队列中出现下面的顺序：

```text
A1 -> A2 -> B1 -> C1
```

如果 `A1` 正在执行，`A2` 因为 session A 的 lane 被占用而暂时不能执行，那么在严格 FIFO 模型下，`B1` 和 `C1` 也会被 `A2` 挡住。这不是 RabbitMQ 的 bug，而是 FIFO 队列的自然结果。只要 worker 只能按队头消费，就无法真正跳过热点 session。

因此问题的本质不是“RabbitMQ 能不能分区”，而是系统是否具备一个任务调度层：它可以识别哪些 session 当前不可执行，把这部分任务暂存在 pending / blocked 视图中，同时继续调度其他可执行 session 的任务。

## 方案一：Redis 分布式锁

Redis 分布式锁是当前阶段最直接的方案。每个 `agent_inbound` 任务执行前，根据 `session_key` 生成锁 key，例如 `lock:agent_inbound:{session_key}`，worker 使用 `SET key value NX EX ttl` 获取执行权。锁的 value 不能只是一个简单标记，而应该包含 worker_id、task_id 或 owner token。释放锁时必须先校验 value，只有当前 worker 仍然是 owner 时才允许释放，避免旧 worker 在卡顿恢复后误删新 worker 的锁。

这个方案的价值在于成本低、落地快，并且能马上解决最危险的问题：同一 session 被多个 worker 同时执行。Redis 的 TTL 也天然适合处理 worker 崩溃后的恢复，worker 正常执行时续租，worker 崩溃后锁自动过期，其他 worker 可以接管。

它的限制也很明确。Redis 锁主要解决“互斥”，不天然保证严格 FIFO。比如 A1、A2、A3 的执行权可以被锁保护，但如果任务重试、重新投递或多个 worker 抢占策略不当，仍然需要任务状态层配合保证顺序。长模型调用还需要续租机制，否则执行时间超过 TTL 后可能出现误接管。Redis 不可用时，也必须明确降级策略：单机模式可以继续运行，多实例模式下则不应该无保护地继续并发执行。

## 方案二：PostgreSQL Advisory Lock

PostgreSQL advisory lock 也可以用来做 session 级互斥。执行任务前，worker 根据 `session_key` 哈希获取数据库 advisory lock，执行完成后释放。如果 worker 崩溃，数据库连接断开后锁会自动释放。这个方案和 PostgreSQL 任务主存储天然贴合，不需要引入额外的 Redis 锁语义。

但它的问题是会把长时间模型调用和数据库连接绑定在一起。Agent 任务可能持续几十秒甚至数分钟，如果大量 worker 长时间持有 advisory lock，就会增加数据库连接管理和锁观测复杂度。PostgreSQL 更适合作为事实存储和状态机落库，不适合作为所有高频临时协调的承载层。因此 advisory lock 可以作为备选方案，但不如 Redis 锁适合作为当前多实例协调的主方案。

## 方案三：RabbitMQ 按 Session 分区队列

RabbitMQ 分区队列的思路是按照 `session_key` 计算分区，把同一个 session 的任务固定路由到同一个 partition queue：

```text
partition = hash(session_key) % N
agent_gateway.inbound.partition.{partition}
```

这种方式比普通竞争消费更接近顺序执行，因为同一 session 的消息天然进入同一个队列。如果每个分区只允许一个 active consumer，并且 `prefetch` 设置得足够保守，那么队列内 FIFO 能提供较强的顺序约束。

不过这个方案有一个关键缺陷：它的调度粒度是 partition，不是 session。一个 partition 里可能同时有 session A、B、C。如果 A 是热点 session 或长任务，A 的后续消息排在队头时，B 和 C 也可能被一起拖慢。也就是说，RabbitMQ 分区可以减少乱序和跨 worker 抢占，但不能彻底解决热点 session 的队头阻塞。

分区队列还会带来拓扑管理成本。分区数量需要提前规划，扩容后 hash 重分布可能让同一 session 的新旧消息落到不同分区。失败重试、死信、重新投递和幂等消费也必须重新设计。因此 RabbitMQ 分区适合中期提升吞吐和队列治理能力，但不应该被理解为最终的 session 调度模型。

## 方案四：任务调度层 Per-session Lane

Per-session lane 是更长期的目标形态。它的核心不是把 RabbitMQ 分区换个名字，而是把“消息排队”和“任务调度”拆开。RabbitMQ 负责可靠分发和唤醒，PostgreSQL 负责保存任务事实，Redis 或数据库状态负责 lane ownership，真正决定任务是否执行的是调度层。

在这个模型下，每个 session 都对应一个逻辑 lane。同一 lane 内任务串行，不同 lane 之间可以并行。假设当前任务池中有：

```text
A1 -> A2 -> B1 -> C1
```

如果 A1 正在执行，A2 不会继续霸占全局队头，而是被识别为当前不可执行任务，留在 pending / blocked 视图中。调度器会继续寻找其他可执行任务，例如 B1 和 C1。等 A1 完成、session A 的 lane 释放后，A2 再被重新调度。这里的 pending bucket 不是一个神秘组件，它本质上是任务事实存储中 `pending` / `retrying` 记录的一种调度视图，再加上运行时计算出的 `blocked_session_keys`。在当前代码里，`TaskWorkerRuntime._blocked_session_keys()` 已经具备这个雏形，它会收集当前因 lane 或 session lock 被阻塞的 session，并在 `LocalTaskQueue.reserve()` 时跳过这些 session。

这个模型的优势是调度粒度真正下降到了 session。热点 session 仍然会阻塞自己，但不应该拖慢其他 session。它也更适合后续支持优先级、取消、重试、限流、迁移、worker 崩溃接管和 Dashboard 观测。但它的实现成本最高，需要明确 lane ownership、续租、接管、任务回流、blocked 视图、幂等和恢复策略。

## RabbitMQ 分区与 Per-session Lane 的本质区别

RabbitMQ 分区队列解决的是“消息如何可靠排队和分发”。它是物理队列层概念，关注 exchange、routing key、queue、ack、nack、prefetch 和 dead-letter。它能把同一 session 的消息尽量放到同一条队列里，但它不知道某个 session 当前是否正在执行，也不知道某个 worker 是否持有这个 session 的执行权。

Per-session lane 解决的是“同一个 session 如何跨 worker 保持互斥、串行、续租、接管和观测”。它是逻辑执行权概念，关注的是 session 当前是否有 owner、owner 是否仍然有效、后续任务是否应该等待、其他 session 是否可以继续调度。

两者不是替代关系，而是分层关系。RabbitMQ 可以作为入站任务的可靠分发层，Redis / PostgreSQL 维护 session lane 的执行权，TaskWorkerRuntime 或后续 scheduler 根据 lane 状态选择可执行任务。最终系统应该是“RabbitMQ 管流动，Redis 管协调，PostgreSQL 管事实，scheduler 管先执行谁”。

## 推荐路线

当前最稳妥的路线是分阶段推进。短期先使用 Redis 分布式锁保护 `agent_inbound` 执行入口，确保同一 session 不会被多个 worker 并发处理。这个阶段的目标不是实现完美 FIFO，而是优先保护会话一致性。实现上应使用 owner token、TTL 和续租机制，并把锁冲突、续租失败、stale lane 等事件写入事件流或 Dashboard。

中期可以引入 RabbitMQ session 分区队列，让入站任务具备更清晰的 broker 拓扑和削峰能力。RabbitMQ 消息仍应只携带 task_id、session_key、partition、idempotency_key 等轻量引用，完整任务事实继续保存在 PostgreSQL。这样即使 RabbitMQ 堆积或重投，也不会把完整用户上下文压在 broker 里。

长期则应把 per-session lane 提升为任务调度核心。这个阶段不再依赖单一 FIFO 队列决定执行顺序，而是由调度器从任务事实存储中选择当前可执行任务。被热点 session 阻塞的任务保留在 pending / blocked 视图中，其他 session 的任务继续执行。这样才能真正解决 `A1 -> A2 -> B1 -> C1` 场景下的队头阻塞问题。

## 结论

当前阶段推荐采用：

```text
Redis session lock / lane ownership + PostgreSQL task 状态
```

它能以较低成本解决最危险的并发问题。RabbitMQ 分区队列适合作为中期增强，用来提升入站任务分发、削峰和死信治理能力，但它不是最终的 session 调度答案。真正要避免热点 session 拖慢其他 session，需要演进到 per-session lane scheduler，让队列只负责分发，lane 负责执行资格，scheduler 负责选择可运行任务，PostgreSQL 负责保存事实状态。
