# 个人饮食减肥 Agent 实施方案

本文档用于后续实现一个面向个人用户的饮食、热量统计和减肥计划 Agent。目标不是做医疗诊断，而是提供可持续、可追踪、可调整的生活方式辅助。所有建议必须提示用户结合自身健康状况判断，涉及疾病、孕期、药物、极端节食、进食障碍风险时应建议咨询医生或营养师。

## 目标形态

系统为每个用户维护独立的个人档案、饮食记录、目标计划和历史反馈。Agent 每天早上 6 点生成当天饮食计划，饭点前后主动提醒用户记录吃了什么，晚上 10 点汇总当天热量摄入、蛋白质和饮食结构，并根据连续数据调整建议。用户可以随时通过企业微信或飞书补录饮食、修改体重目标、询问替代食物或查询近期趋势。

该 Agent 不应依赖全局记忆混写。个人信息必须按用户隔离，建议以 PostgreSQL 作为主存储，本地 JSONL 作为审计和回放兜底。Redis 可用于提醒去重、当日状态缓存和短期幂等，RabbitMQ 继续承担主动任务投递和异步执行。

## 为什么需要数据库持久化个人信息

需要使用数据库持久化。原因是饮食减肥 Agent 的核心不是一次性回答，而是跨天、跨周持续跟踪。用户的身高、体重、目标体重、饮食偏好、忌口、运动频率、工作作息、历史摄入、体重变化和计划执行情况都需要被稳定查询、统计和修正。仅依赖 Markdown 或 memory_write 会带来三个问题：结构不稳定、统计困难、用户隔离弱。

PostgreSQL 适合承担主存储，因为它可以同时保存结构化记录和 JSONB 扩展字段。结构化字段用于热量统计、时间范围查询和趋势分析，JSONB 用于保留模型抽取的食物明细、置信度、原始文本和估算依据。MySQL 也能实现，但当前 Gateway 已经有 PostgreSQL 状态仓储、schema 初始化、read/write repository 和 Docker Compose，因此优先沿用 PostgreSQL 成本最低。

## 用户身份与隔离

用户主键建议使用统一 `user_scope`，格式与当前记忆系统保持一致。对企业微信用户可以使用 `user:wework:{account_id}:direct:{user_id}`，对飞书用户可以使用 `user:feishu:{account_id}:direct:{open_id}`。这个值来自入站消息路由后的 session key 派生，也可以在通道层显式写入 metadata。

所有个人数据表都必须包含 `user_scope` 字段。所有查询默认必须带 `user_scope`。Dashboard 可以展示聚合数量，但不能把不同用户的饮食明细混在一起。Agent 工具层也需要自动注入当前用户 scope，避免模型忘记传参造成串号。

## 数据模型

第一阶段可以新增五类表。

`user_profiles` 保存用户基本档案。字段包括 `user_scope`、`display_name`、`gender`、`birth_year`、`height_cm`、`current_weight_kg`、`target_weight_kg`、`activity_level`、`timezone`、`diet_preferences`、`allergies`、`medical_notes`、`created_at`、`updated_at`、`metadata`。

`weight_logs` 保存体重记录。字段包括 `id`、`user_scope`、`weight_kg`、`recorded_at`、`source`、`metadata`。它用于生成每周趋势和调整热量目标。

`meal_logs` 保存每餐记录。字段包括 `id`、`user_scope`、`meal_date`、`meal_type`、`raw_text`、`items`、`estimated_calories`、`protein_g`、`carbs_g`、`fat_g`、`confidence`、`logged_at`、`metadata`。`items` 使用 JSONB 保存食物、份量、估算热量和依据。

`daily_nutrition_summaries` 保存每日汇总。字段包括 `id`、`user_scope`、`date`、`target_calories`、`actual_calories`、`protein_g`、`carbs_g`、`fat_g`、`summary_text`、`risk_flags`、`created_at`、`updated_at`、`metadata`。

`diet_plans` 保存计划。字段包括 `id`、`user_scope`、`plan_date`、`target_calories`、`meals`、`shopping_tips`、`generated_reason`、`status`、`created_at`、`metadata`。`meals` 使用 JSONB 保存早餐、午餐、晚餐和加餐建议。

## 工具设计

需要新增一组工具，全部自动注入 `user_scope`。

`profile_get` 查询用户档案，如果不存在，返回需要补充的信息。`profile_update` 更新身高、体重、目标、偏好、忌口和作息。`meal_log_add` 从用户自然语言中保存一餐记录，可以先让模型抽取结构化 JSON，再由工具落库。`meal_log_list` 查询当天或指定日期饮食记录。`nutrition_day_summary` 统计当天热量和宏量营养素。`diet_plan_generate` 根据档案、目标、历史执行情况生成当天计划。`weight_log_add` 保存体重。`progress_summary` 输出近 7 天、近 30 天趋势。

这些工具不应该让模型直接拼 SQL。工具层负责参数校验、默认日期、用户隔离、数据库写入和异常处理。模型只负责解释用户输入、提出建议和组织表达。

## 定时任务设计

每天 6 点生成当天饮食计划。Cron payload 使用 `agent_turn` 或更推荐新增专用 `diet_plan_generate` task type。专用 task type 更稳定，因为它可以先读数据库生成结构化计划，再交给 Agent 润色，最后通过可靠投递发给用户。

饭点提醒建议配置多个 Cron。早餐前 7:30 提醒“今天早餐计划是……吃完后回复实际吃了什么”。早餐后 9:30 如果当天早餐未记录，提示补录。午餐前 11:30、午餐后 13:30、晚餐前 17:30、晚餐后 20:00 同理。提醒任务必须按 `user_scope + date + meal_type + reminder_stage` 做幂等，避免多实例重复提醒。

每天 22 点生成当天摄入汇总。任务读取当天 meal_logs，计算总热量、蛋白质、碳水、脂肪和执行偏差。如果缺少餐次，应明确指出“缺少午餐记录，估算不完整”，不要假装准确。

## 交互流程

首次使用时，Agent 不应直接给复杂计划，而是先收集最少必要信息：身高、体重、目标体重、性别或基础代谢估算偏好、日常活动水平、忌口、常见用餐地点、是否有疾病或特殊饮食限制。用户可以跳过部分信息，系统用保守默认值并标记置信度较低。

用户回复“中午吃了牛肉饭一份、无糖可乐”时，Agent 调用 `meal_log_add`，保存原文和估算结果，然后回复“已记录，估算约 X kcal，主要来自米饭和牛肉；如果份量偏大可以告诉我修正”。用户回复“今天跑步 30 分钟”时，后续可以扩展 `activity_log_add`，但第一版可以先写入备注或记忆，不参与精准热量抵扣。

晚上汇总时，Agent 输出应包含四部分：今日总摄入、与目标差值、做得好的地方、明天可执行的一条调整建议。避免输出太长，也避免道德评价。

## 热量估算策略

第一版采用“模型估算 + 结构化保存 + 置信度”的方式，不追求医学级精确。工具保存每个食物项时记录 `name`、`amount_text`、`calories`、`protein_g`、`confidence` 和 `reason`。如果用户描述模糊，例如“一碗面”，置信度应低，并提示可通过补充份量、照片（暂不支持多模态识别）或品牌提升准确度。

后续可以接入食物热量库。优先从本地常见食物表开始，例如米饭、面条、鸡蛋、鸡胸肉、牛肉、奶茶、咖啡、外卖常见套餐。再考虑第三方 API。为了稳定性，外部 API 不应阻塞主流程，查不到时退回模型估算。

## 减肥计划原则

计划必须实用，不能极端。默认热量缺口建议控制在每日 300 到 500 kcal，优先保证蛋白质和饱腹感。不要建议极低热量饮食，不要承诺快速减重。计划应考虑用户偏好，比如常吃外卖、没有厨房、预算有限、晚睡、工作日忙等。

输出计划时使用“可执行选项”而不是硬性菜单。早餐给 2 个选项，午餐给 2 个选项，晚餐给 2 个选项，并给出替换规则。例如“主食半拳到一拳，蛋白质一掌，蔬菜两拳”。这样比固定菜谱更容易坚持。

## Agent Prompt 设计

Agent 身份建议是“个人饮食与体重管理助手”。系统提示词应强调：先记录事实，再给建议；不做医疗诊断；估算要标注不确定性；优先简单可执行；长期目标以周为单位调整；不要因为一天超标就给过度建议。

记忆层只保存长期偏好和稳定事实，例如“不吃香菜”“早餐通常在公司吃”“喜欢中餐”“当前目标是从 82kg 到 75kg”。每日饮食记录不要写入普通 memory，而应写入 meal_logs 表，避免记忆系统膨胀。

## 实施阶段

第一阶段完成用户档案和饮食记录表、工具、企业微信交互闭环。完成标准是用户能通过企业微信补充档案、记录一餐，并在晚上收到当天统计。

第二阶段完成定时任务。完成标准是每天 6 点、饭点前后、22 点能对指定用户发送计划、提醒和汇总，且多实例下不会重复发送。

第三阶段完成计划个性化。完成标准是计划能根据用户历史记录、偏好、目标和最近执行情况调整，例如连续三天晚餐超标后给出更轻量的晚餐替代方案。

第四阶段完成 Dashboard 和控制面。完成标准是可以查看用户档案、最近饮食记录、今日热量、缺失餐次、定时任务执行状态和投递状态。

第五阶段再考虑图片识别、食物库、运动记录、体重趋势图和周报。

## 风险与边界

最大风险是错误估算给用户造成误导。因此每次估算都需要保留原文、置信度和可修正入口。第二个风险是隐私，个人体重和饮食数据必须按 `user_scope` 隔离，不能出现在其他用户上下文中。第三个风险是提醒过多造成打扰，所以每个用户需要可配置提醒开关、饭点时间和免打扰时间。

如果用户提供疾病、药物、孕期、进食障碍、极端节食等信息，Agent 应停止给具体减重方案，改为建议咨询专业人士，并只提供通用记录和温和提醒。
