# 用例图绘制规范

## 适用范围

用例图用于表达系统对外提供的功能、外部参与者以及参与者与用例之间的关系。论文中如果需要说明“谁使用系统完成什么目标”，应使用用例图；如果要说明内部调用顺序，应使用时序图；如果要说明实现类结构，应使用类图。

## 权威依据

- OMG UML 2.5.1 UseCases 相关规范。来源：https://www.omg.org/spec/UML/2.5.1/

## 基本元素

- 参与者 Actor：系统外部与系统交互的人、组织、外部系统或设备。默认使用 UML 小人符号，不使用矩形框代替。
- 用例 Use Case：椭圆，名称使用用户目标或系统能力，如“登录系统”“重置密码”。
- 系统边界 Subject：矩形边界，内部放系统负责的用例，边界标题写系统名。
- 关联 Association：参与者与用例之间的实线，表示参与者参与该用例。
- 包含 Include：虚线开放箭头，标注 `<<include>>`，从基础用例指向必然执行的被包含用例。
- 扩展 Extend：虚线开放箭头，标注 `<<extend>>`，从扩展用例指向被扩展用例，表示条件性扩展行为。
- 泛化 Generalization：实线空心三角箭头，子参与者或子用例指向父参与者或父用例。

## 绘制约束

1. 只画系统边界外部可见的目标，不画按钮点击、接口调用、SQL 查询等内部步骤。
2. 用例名称应是动宾短语或目标短语，不用技术实现名，如“验证密码哈希”通常不是对外用例。
3. 参与者必须在系统边界外；用例必须在系统边界内。
4. 普通关联线不加箭头；`include`、`extend`、泛化必须使用正确方向。
5. `include` 用于必然复用的子用例；`extend` 用于可选或异常扩展，不要为了“看起来完整”滥用。
6. 避免交叉线。参与者放在边界左右两侧，用例按主次分组。
7. 用例图不要表达时间顺序。登录步骤的先后顺序应放到流程图或时序图。

## 论文 draw.io 样式

- 系统边界：白底或无填充黑色矩形，标题置于左上角。
- 用例：白底黑边椭圆，`fontFamily=SimSun`，`fontSize=12`。
- 参与者：使用 UML actor 小人符号；只有在用户明确要求纯文本兼容时，才使用白底黑边矩形并标注 `<<actor>>`。
- 线条：黑色 1 px。普通关联使用无箭头实线；`include`/`extend` 使用虚线开放箭头。
- 布局：整体横向展开，系统边界位于中间，主要参与者在左侧，外部系统在右侧。

## 常见错误

- 把系统内部步骤画成用例。
- 参与者放在系统边界内部。
- `include` 和 `extend` 箭头方向反了。
- 用例之间连线过多，导致图像变成流程图。
- 普通参与关系使用了箭头，暗示错误的控制方向。
