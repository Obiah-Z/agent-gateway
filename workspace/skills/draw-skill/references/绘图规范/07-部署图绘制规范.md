# 部署图绘制规范

## 适用范围

部署图用于表达软件制品、运行环境、计算节点、网络边界和部署关系。论文中若要说明系统如何运行在服务器、容器、Kubernetes、云资源或边缘设备上，应使用部署图。

## 权威依据

- OMG UML 2.5.1 Deployment Diagram / Node / Artifact / DeploymentSpecification 相关规范。来源：https://www.omg.org/spec/UML/2.5.1/
- C4 Model Deployment Diagram：可作为软件架构部署视图的补充组织方式。来源：https://c4model.com/diagrams/deployment

## 基本元素

UML 部署图：

- Node：运行资源，通常是设备、执行环境、服务器、虚拟机、容器平台等。
- Execution Environment：执行环境，如 JVM、Docker、Kubernetes Namespace。
- Artifact：部署制品，如 jar、镜像、前端静态包、配置文件。
- Deployment：制品部署到节点或执行环境。
- Communication Path：节点之间通信路径，可标注协议、端口或网络。

C4 部署图：

- Deployment Node：物理或虚拟基础设施节点，可嵌套。
- Container Instance：容器/服务实例。
- Infrastructure Node：负载均衡、数据库、消息队列、DNS、CDN 等。
- Relationship：运行时通信关系。

## 绘制约束

1. 明确部署环境名称，如“生产环境”“测试环境”“边缘节点”。
2. 节点可嵌套，但层级不宜超过三层：区域/网络、主机或集群、执行环境/实例。
3. Artifact 与逻辑服务要区分：`order-service.jar` 是制品，`订单服务` 是服务实例或容器。
4. 通信路径必须标注关键协议或端口，如 HTTPS、gRPC、MySQL、Redis。
5. 副本数、资源规格、命名空间等只有在论文论证需要时才标注。
6. 不在部署图中展开类、数据库字段或业务流程。
7. 若使用 Kubernetes，画清 Ingress/Service/Pod/Deployment 的抽象层次，不必画所有底层对象。

## 论文 draw.io 样式

- 节点：白底黑边三维盒或普通矩形；建议普通矩形嵌套更清晰。
- 执行环境：节点内部的白底黑边矩形，标注 `<<executionEnvironment>>`。
- 制品：白底黑边矩形，标注 `<<artifact>>`。
- 通信路径：黑色 1 px 实线，箭头方向表示主要访问方向。
- 字体：宋体 12 pt；节点标题可加粗。
- 网络边界或区域边界使用黑色虚线大框。

## 常见错误

- 把逻辑架构图当成部署图，没有运行节点。
- 只写“服务器”而不说明部署了什么。
- 协议和访问路径缺失。
- 把业务流程步骤放到部署节点里。
- Kubernetes 图过细，导致论文中无法阅读。
