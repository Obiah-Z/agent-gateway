# 活动图绘制规范

## 适用范围

活动图用于表达一个业务过程或系统行为中的控制流、分支、合并、并发和结束条件。论文中如果需要说明“登录认证如何一步步完成，并在不同条件下走向成功或失败”，应使用活动图；如果只需要传统程序步骤，可使用流程图；如果要表达对象之间的消息交互顺序，应使用时序图；如果要表达对象生命周期，应使用状态图。

## 权威依据

- OMG UML 2.5.1 Activity、Action、ActivityNode、ControlNode、ActivityEdge、ActivityPartition 相关规范。来源：https://www.omg.org/spec/UML/2.5.1/

## 基本元素

- 初始节点 Initial Node：实心黑圆，表示活动开始。
- 活动终止节点 Activity Final：黑色实心圆外加空心圆，表示整个活动结束。
- 动作 Action：圆角矩形，表示一个不可再分或当前抽象层级下不再展开的行为。
- 判断节点 Decision Node：菱形，一个入口、多个带守卫条件的出口。
- 合并节点 Merge Node：菱形，多个入口、一个出口，用于合并互斥路径。
- 分叉节点 Fork Node：粗横条或竖条，一个入口、多个出口，表示并行开始。
- 汇合节点 Join Node：粗横条或竖条，多个入口、一个出口，表示并行同步。
- 控制流 Control Flow：带箭头实线，表示动作执行顺序。
- 对象流 Object Flow：带箭头实线，可连接对象节点；仅当论文需要强调数据对象传递时使用。
- 活动分区 Activity Partition：可选泳道，用于区分用户、系统、外部服务等职责边界。

## 绘制约束

1. 活动图表达行为流，不表达对象结构、数据库字段或部署节点。
2. 动作名称应使用短动宾短语，如“输入账号和密码”“查询用户”“创建会话”。
3. 判断节点的每条出口必须标注守卫条件，如 `[是]`、`[否]`、`[输入合法]`。
4. 多个异常路径可以分别指向错误提示动作，再汇入统一结束节点；不要让错误线穿过主路径节点。
5. 有并行业务时使用分叉/汇合条；没有并行语义时不要用粗条装饰流程。
6. 分区只在职责划分对论文论点有帮助时使用。简单登录流程可不使用泳道，以保持图面紧凑。
7. 不要把活动图画成时序图：活动图中不需要参与者生命线和消息箭头。
8. 不要把活动图画成状态图：节点表示动作或控制节点，不表示对象长期状态。

## 论文 draw.io 样式

- 初始节点：`shape=startState`，黑色填充，黑色边框。
- 活动终止节点：`shape=endState`，黑色填充，黑色边框。
- 动作：白底黑边圆角矩形，`fontFamily=SimSun`，`fontSize=12`，尺寸贴合文本。
- 判断/合并：白底黑边菱形，条件文字居中，出口守卫标签靠近对应连线。
- 分叉/汇合：黑色窄矩形，横向流程中优先使用竖条；纵向流程中优先使用横条。
- 连线：黑色 1 px，正交连接，箭头指向后续动作。
- 布局：简单流程优先横向展开；分支动作放在判断节点上方或下方，避免分支线穿过其他节点。
- 颜色：论文图默认只使用黑、白、灰，不使用彩色区分成功/失败。

## 常见错误

- 判断出口没有守卫条件。
- 把用户、服务、数据库画成一列列生命线，误画成时序图。
- 用普通矩形代替 UML 初始/终止节点。
- 分支线穿过动作节点或错误提示节点。
- 把“登录成功”“登录失败”等结果都画成状态，而没有明确动作与终止节点。
