# 架构图绘制规范

## 适用范围

架构图用于表达软件系统的主要构成、职责边界、外部依赖和组件关系。论文中推荐采用 C4 模型组织架构图：系统上下文图、容器图、组件图，必要时补充动态图。

## 权威依据

- Simon Brown, C4 Model 官方网站：软件系统、容器、组件、代码四级抽象；系统上下文图、容器图、组件图、代码图及动态图、部署图等支持图。来源：https://c4model.com/
- UML 2.5.1 Component Diagram 可用于更形式化的组件接口与依赖表达。来源：https://www.omg.org/spec/UML/2.5.1/

## 基本元素

C4 常用抽象：

- Person：使用系统的人或角色。
- Software System：一个完整的软件系统。
- Container：可独立运行或部署的应用、服务、数据库、文件系统等。
- Component：容器内部承担明确职责的组件。
- Relationship：元素之间的依赖、调用、数据流或消息。

UML 组件图常用元素：

- Component：带组件标识的矩形或 `<<component>>`。
- Provided Interface：提供接口。
- Required Interface：所需接口。
- Dependency：虚线开放箭头。

## 绘制约束

1. 先确定图层级。上下文图只画系统和外部人/系统；容器图只画系统内部主要容器；组件图只画单个容器内部组件。
2. 每个元素必须写名称、类型和简短职责。C4 常见写法为“名称 / 类型 / 说明”。
3. 关系线必须写明语义，例如“调用 REST API”“发布事件”“读取订单数据”。
4. 明确系统边界。被研究系统用边界框包住，外部系统放在边界外。
5. 架构图不表达精确时序；需要时序时另画时序图。
6. 不把部署节点、Pod、副本数、IP 地址混入逻辑架构图；这些属于部署图。
7. 若图中有数据库、消息队列、缓存，应作为容器或基础设施依赖明确标注。

## 论文 draw.io 样式

- C4 元素：白底黑边矩形或圆角矩形；同一层级形状一致。
- 系统边界：白底或无填充黑色虚线/实线大矩形，标题置于左上角。
- 关系线：黑色 1 px，开放或实心箭头均可，但全图统一。
- 数据库：可使用圆柱形，但保持黑白。
- 字体：宋体 12 pt；元素说明可用 10 pt。
- 不使用 C4 常见彩色主题，论文中以黑白线稿为准。

## 常见错误

- 一张图同时画上下文、容器、组件和部署细节。
- 关系线没有说明。
- 系统边界不清，读者不知道哪些属于研究对象。
- 元素命名是技术缩写，缺少职责说明。
- 颜色承载语义，黑白打印后不可理解。
