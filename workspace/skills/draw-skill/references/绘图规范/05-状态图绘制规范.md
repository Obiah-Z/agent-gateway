# 状态图绘制规范

## 适用范围

状态图用于表达一个对象、实体或系统在生命周期内的状态、事件触发和状态迁移。适合论文中说明订单、任务、会话、设备、协议连接等状态机。

## 权威依据

- OMG UML 2.5.1 StateMachine / State / Transition 相关规范。来源：https://www.omg.org/spec/UML/2.5.1/

## 基本元素

- 初始状态：实心黑圆。
- 终止状态：带外圈的实心黑圆。
- 状态：圆角矩形，写状态名；复杂状态可包含 entry/do/exit 行为。
- 转换：有向边，格式为 `事件 [守卫] / 动作`。
- 复合状态：状态框内嵌子状态。
- 并发区域：复合状态内用虚线分隔区域。
- 历史状态：浅历史 `H`，深历史 `H*`。

## 绘制约束

1. 状态名使用稳定状态名词或形容词短语，如“待支付”“已取消”。
2. 转换标签按 UML 顺序书写：触发事件、守卫条件、动作。
3. 每个状态图聚焦一个对象或系统，不同时画多个对象的生命周期。
4. 初始状态只能表示进入状态机的起点，不表示业务步骤。
5. 终止状态用于生命周期结束；若对象持续存在，可不画终止状态。
6. 分支条件要互斥且覆盖主要路径，异常路径应清晰标注。
7. 不把状态图画成流程图；状态是“持续情形”，不是“一次动作”。

## 论文 draw.io 样式

- 初始/终止：黑色标准 UML 符号。
- 状态：白底黑边圆角矩形。
- 转换：黑色 1 px 实线箭头；标签靠近线段中部。
- 复合状态：外层白底黑边大圆角矩形，内部子状态留足边距。
- 字体：宋体 12 pt。

## 常见错误

- 状态名写成动词，如“创建订单”“执行支付”。
- 转换线没有触发事件或条件。
- 初始状态连到多个状态但没有选择伪状态或条件。
- 把活动步骤按顺序排列成状态图。
- 终止状态被当成普通“结束步骤”。
