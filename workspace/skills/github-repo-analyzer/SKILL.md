# GitHub 仓库分析

当用户给出 GitHub 仓库链接，并要求解释“这个项目是干什么的”“值不值得看”“怎么运行”“有什么可借鉴点”时，使用本技能。

## 默认模式

默认执行标准版分析：

- 可以读取 GitHub 页面、README、文档、目录结构、配置文件、示例、测试和核心源码入口。
- 可以克隆仓库到临时目录做只读扫描。
- 不默认安装依赖、运行脚本、启动服务、执行测试或调用仓库内代码。
- 如果用户要求运行项目或测试，先说明这是深度版动作，可能执行不可信代码，需要明确确认。

## 输出落地

每次分析都必须写入 Markdown 文件：

```text
workspace/reports/github-repos/仓库分析-{owner}-{repo}.md
```

命名规则：

- 使用中文前缀 `仓库分析-`。
- `{owner}` 和 `{repo}` 来自仓库链接。
- 空格、下划线、点号和特殊字符统一转为短横线。
- 文件名全部小写，避免覆盖无关文件。

对话回复只给简短摘要和文档路径，不要把完整报告刷屏。

## 标准版分析流程

1. 解析仓库链接，确认 `owner/repo`。
2. 收集仓库基本信息：地址、主要语言、License、Star、Fork、最近更新时间、默认分支。
3. 读取 README、docs、examples、tests、配置文件和核心源码入口。
4. 识别项目类型：框架、工具库、应用、模板、Agent 项目、研究项目或其他。
5. 用自己的话解释项目目标、核心功能、技术栈和主流程。
6. 判断本地运行路径，只根据仓库文档和配置推导，不编造步骤。
7. 提取值得重点阅读的文件，并说明阅读理由。
8. 结合 AI Agent Gateway 视角提炼可借鉴点。
9. 输出风险、不足、成熟度评分和最终建议。
10. 写入 Markdown 报告。

## 报告模板

```markdown
# GitHub 仓库分析：owner/repo

## 1. 项目结论

一句话说明项目是干什么的，以及是否值得继续看。

## 2. 基本信息

- 仓库地址：
- 主要语言：
- License：
- Star / Fork：
- 最近更新时间：
- 默认分支：
- 项目类型：

## 3. 项目解决什么问题

说明目标用户、使用场景和核心痛点。

## 4. 核心功能

列出 3 到 8 个真实功能点，优先来自 README、示例、文档和源码入口。

## 5. 技术栈与依赖

说明语言、框架、包管理器、运行时、外部服务、模型 API、数据库和部署工具。

## 6. 目录结构解读

解释关键目录和文件分别负责什么。

## 7. 核心流程或架构理解

说明请求如何进入、核心模块如何协作、配置如何加载、数据如何流动、扩展点在哪里。

## 8. 如何本地运行

列出安装依赖、环境变量、启动命令、测试命令和示例入口。无法确认时明确写“未在仓库中找到可靠说明”。

## 9. 值得重点阅读的文件

列出 5 到 10 个文件，并说明为什么值得先看。

## 10. 可借鉴点

结合 Gateway 项目，说明可参考的设计，例如 Agent Loop、Tool Calling、Skill 机制、调度、记忆、Dashboard、错误处理、配置组织和测试方式。

## 11. 风险与不足

从文档完整度、可运行性、维护活跃度、测试覆盖、依赖复杂度、安全风险和过度设计等角度判断。

## 12. 成熟度评分

| 维度 | 评分 | 说明 |
|---|---:|---|
| 文档完整度 | 1-5 | |
| 代码结构清晰度 | 1-5 | |
| 可运行性 | 1-5 | |
| 维护活跃度 | 1-5 | |
| 与当前 Gateway 项目的相关度 | 1-5 | |

## 13. 最终建议

说明是否值得继续研究，最值得看的模块，适合直接使用、Fork、只借鉴思路还是不建议投入，以及下一步建议。
```

## 判断要求

- 不照搬 README 宣传语，要用中文解释实际能力。
- 不能编造 Star、Fork、License、最近更新时间或运行命令。
- 如果来源不足，明确写“未确认”或“仓库中未找到可靠说明”。
- 可借鉴点要贴合 Gateway，不要泛泛写“代码结构清晰”。
- 风险判断要具体，例如缺少测试、依赖重、配置不清、维护停滞、需要外部服务或存在凭据风险。
