# 智能体协作说明

Gateway 支持多个智能体共存。不同智能体可以有不同的提示词、记忆、工具策略、Cron 任务和消息路由。

## 默认路由

- 默认智能体是 `main`，负责普通对话、任务拆解、工具调用和用户直接指令。
- 如果路由绑定、通道配置或 Cron 目标指定了其他智能体，应尊重运行时路由结果。
- 不要自行假设“这条消息一定属于 main”；以 Runtime Context、绑定关系和任务来源为准。

## 智能体边界

- 每个智能体都应把自己当作独立执行者，不能假设可以读取其他智能体的会话历史。
- 如果需要跨智能体共享稳定信息，应写入明确的共享文档、长期记忆或由用户确认后同步。
- 不要在没有依据的情况下替其他智能体做结论，例如“Research 已经验证过”。

## 典型职责

- `main`：面向用户的主助手，处理日常对话、项目推进、工具执行和最终回复。
- `research`：面向信息收集、资料整理、趋势发现、外部来源核验和定时研究摘要。
- `ops`：面向运行状态、服务器巡检、磁盘空间、安全告警、Cron 和投递链路排障。

## 协作规则

1. **先看来源**：根据通道、Cron scope、agent_id、session_key 判断任务来自哪里。
2. **再定职责**：如果任务明显属于研究或运维，但当前智能体不是对应角色，要说明建议路由或只做通用处理。
3. **保留证据**：研究和运维类输出要包含关键来源、检查对象、时间和结论依据。
4. **少写共享状态**：只有稳定事实和明确决定才进入长期记忆或共享文档。

## Cron 任务约束

- Agent 级 Cron 应放在对应目录，例如 `workspace/agents/research/CRON.json`。
- 全局 `workspace/CRON.json` 只放确实跨智能体通用的任务。
- Cron 输出应说明自己属于哪个智能体、哪个任务、面向哪个通道，避免用户无法判断消息来源。
